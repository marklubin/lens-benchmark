**Item: Start doing technical design reviews BEFORE sprint commitment**
Written by: Derek Lane, Angela Fong
Votes: 6 (all team members except Kai who facilitated)

[Derek Lane — 3:14 PM]
The payment reconciliation retry logic got pulled because we underestimated the complexity. When I actually sat down to design it, I realized it needs a state machine with 7 states, idempotency guarantees, and dead-letter handling. That's not a "5 story point" ticket — it's probably 13. If we'd done a design review before sprint planning, we would have sized it correctly and either committed to less or broken it into smaller pieces.

[Angela Fong — 3:15 PM]
Same thing happened with my partner API work two sprints ago. The ticket said "implement rate limiting" but when I dug in, the sliding window algorithm had edge cases around partner timezone boundaries that took 3 extra days. A 30-minute design review would have caught that.

[Marcus Webb — 3:15 PM]
Can you elaborate on the timezone edge case, Angela? I want to make sure we don't hit the same kind of thing again with the usage dashboard.

[Angela Fong — 3:16 PM]
Yeah, so partners in APAC have rate limit windows that reset at midnight their local time, but our Redis keys were using UTC. When the sliding window crossed the UTC midnight boundary, the counter was resetting for some partners mid-window. The fix was to key the windows by partner timezone offset, but that required a schema change in the partner configuration table. That kind of second-order complexity is exactly what a design review catches.

[Priti Sharma — 3:16 PM]
Counter-point: if we do design reviews for every ticket, it adds overhead. We're already spending 15% of our sprint on ceremonies. Can we limit it to tickets above a certain size? Like, anything over 5 story points gets a mandatory design review?

DECISIONS LOG

1. **We decided to require design reviews for tickets estimated at 5+ story points** before sprint commitment. Lightweight design doc (1 page) reviewed by at least one other engineer. Format: problem statement, proposed approach, known risks, open questions. Doc goes on Confluence, linked from Jira ticket.

2. **We decided to adopt mandatory pairing for infrastructure and data pipeline tickets.** Optional for feature work. Infrastructure defined as: databases, caching, deployment configuration, monitoring, shared libraries — anything other services depend on. Pairing sessions scheduled within the 10am–3pm CT overlap window.

3. **We decided to enforce 48-hour post-incident reviews for P2+ incidents.** Reviews block non-emergency deployments until completed. Deployment block applies team-wide to create collective ownership. Template: timeline, root cause (five-whys), impact, action items, owner. Target: 30 minutes per review.

4. **We decided to enforce formal trade-off discussions for all mid-sprint scope additions.** Must identify what's being dropped. P1 exempt, P2 expedited (same-day decision). Scope changes logged on the sprint board for trend tracking.

5. **We chose to ban deployments after 2pm on Fridays.** Morning deployments allowed with 2-hour monitoring requirement. CI/CD pipeline will enforce the cutoff with a manual override option for Kai or Derek.

6. **We decided to continue async standups and add a weekly 15-minute Wednesday sync standup.** Scheduled at 10:30 AM CT. Agenda: blockers, cross-cutting issues, and anything that doesn't fit in a Slack post.

7. **Continuing the 24-hour code review SLA and sprint demos.** No changes. Will revisit SLA escalation rules if breach frequency increases.
