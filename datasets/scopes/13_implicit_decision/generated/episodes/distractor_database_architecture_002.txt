[Hannah Choi — 10:10 AM]
**Option 1: TimescaleDB**

TimescaleDB is a PostgreSQL extension for time-series data. It adds "hypertables" — automatically partitioned tables where each partition (called a "chunk") covers a time range. Queries that specify a time range only scan the relevant chunks.

Pros:
- Full SQL support — our team knows SQL, our dashboards use SQL queries, no new query language
- PostgreSQL ecosystem — works with all PostgreSQL tools, extensions, and drivers
- Continuous aggregates — materialized views that auto-update as new data arrives. Perfect for our hourly/daily aggregation tiers
- Compression — 10-20x compression on historical chunks. Our 2.2TB raw data would compress to ~150GB
- We already chose PostgreSQL for InventoryHub v2 (ADR-042), so we have the operational muscle

Cons:
- Single-node performance ceiling. TimescaleDB's multi-node (distributed) was deprecated in favor of their cloud product. For our self-managed deployment, we're limited to what a single large instance can handle.
- Write throughput: benchmarked at 380K rows/sec on our test hardware. Our requirement is 14K rows/sec (121M points / 86,400 seconds). That's 27x headroom. Sufficient.
- Query performance: 24-hour temperature aggregation across 100 sensors — 180ms on compressed data. That's 17x faster than Elasticsearch's 3.2 seconds.

Cost estimate: Single db.r6g.2xlarge (8 vCPU, 64GB RAM) with 500GB gp3 EBS. Monthly cost: ~$1,200. That's $12,800/month savings versus Elasticsearch.

[Hannah Choi — 10:19 AM]
**Option 2: InfluxDB 3.0**

InfluxDB is a dedicated time-series database. Version 3.0 is a ground-up rewrite using Apache Arrow and DataFusion for columnar storage and query execution.

Pros:
- Purpose-built for time-series — ingestion, storage, and query are all optimized for temporal data
- Columnar storage — extremely efficient for aggregation queries (only reads the columns needed)
- InfluxQL and SQL support — familiar query languages
- Built-in retention policies and downsampling — define rules to automatically aggregate and delete old data

Cons:
- InfluxDB 3.0 is relatively new (GA in early 2025). Fewer production deployments than TimescaleDB
- Operational unfamiliarity — nobody on the team has run InfluxDB in production
- Separate system — doesn't share infrastructure with our PostgreSQL stack
- Clustering requires InfluxDB Enterprise (paid) for high availability

[Hannah Choi — 10:29 AM]
**Option 3: ClickHouse**

ClickHouse is a columnar analytics database. Not purpose-built for time-series but excellent at it due to columnar storage and vectorized query execution.

Pros:
- Extreme query performance — the fastest analytical query engine available. Our benchmark: 24-hour aggregation in 12ms. Multi-month trend in 85ms.
- Excellent compression — 15-30x on time-series data
- SQL support with extensions for time-series patterns (tumble, hop, session windows)
- Active open-source community, large company deployments (Uber, Cloudflare, eBay)

Cons:
- Not designed for high-frequency small writes. ClickHouse prefers batched inserts (1000+ rows per insert). Our sensors emit every 10 seconds — we'd need a write buffer (Kafka or in-memory) to batch writes.
- No built-in retention policies — we'd implement them as scheduled DROP PARTITION operations
- Updates and deletes are expensive (ClickHouse is append-optimized)
- Operational complexity — ClickHouse cluster management is non-trivial. ZooKeeper dependency for replication.

SCHEMA DESIGN PREVIEW (10:55 AM — 11:30 AM)

[Hannah Choi — 11:06 AM]
Quick preview of the schema. I'll do a full design review next week but I want to validate the hypertable structure. Let me share my screen again — here's the SQL.

```sql
-- Enable TimescaleDB extension
CREATE EXTENSION IF NOT EXISTS timescaledb;

-- Raw telemetry data
CREATE TABLE sensor_telemetry (
    time TIMESTAMPTZ NOT NULL,
    sensor_id UUID NOT NULL,
    warehouse_id UUID NOT NULL,
    sensor_type VARCHAR(30) NOT NULL,  -- temperature, humidity, door, forklift, energy
    value_numeric DOUBLE PRECISION,     -- for temperature, humidity, energy
    value_text VARCHAR(100),            -- for door (open/close), forklift (zone entry/exit)
    metadata JSONB                      -- sensor-specific additional data
);

-- Convert to hypertable with 1-day chunks
SELECT create_hypertable('sensor_telemetry', 'time',
    chunk_time_interval => INTERVAL '1 day'
);

-- Indexes
CREATE INDEX idx_telemetry_sensor_time ON sensor_telemetry (sensor_id, time DESC);
CREATE INDEX idx_telemetry_warehouse_type_time ON sensor_telemetry (warehouse_id, sensor_type, time DESC);

-- Compression policy: compress chunks older than 7 days
ALTER TABLE sensor_telemetry SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'sensor_id, sensor_type',
    timescaledb.compress_orderby = 'time DESC'
);
SELECT add_compression_policy('sensor_telemetry', INTERVAL '7 days');

-- Retention policy: drop raw data older than 90 days
SELECT add_retention_policy('sensor_telemetry', INTERVAL '90 days');

-- Continuous aggregate: hourly averages (retained 2 years)
CREATE MATERIALIZED VIEW sensor_hourly
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 hour', time) AS bucket,
    sensor_id,
    warehouse_id,
    sensor_type,
    AVG(value_numeric) AS avg_value,
    MIN(value_numeric) AS min_value,
    MAX(value_numeric) AS max_value,
    COUNT(*) AS sample_count
FROM sensor_telemetry
WHERE value_numeric IS NOT NULL
GROUP BY bucket, sensor_id, warehouse_id, sensor_type;

-- Refresh policy: refresh hourly aggregates every hour
SELECT add_continuous_aggregate_policy('sensor_hourly',
    start_offset => INTERVAL '3 hours',
    end_offset => INTERVAL '1 hour',
    schedule_interval => INTERVAL '1 hour'
);

-- Continuous aggregate: daily averages (retained 5 years)
CREATE MATERIALIZED VIEW sensor_daily
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 day', bucket) AS bucket,
    sensor_id,
    warehouse_id,
    sensor_type,
    AVG(avg_value) AS avg_value,
    MIN(min_value) AS min_value,
    MAX(max_value) AS max_value,
    SUM(sample_count) AS sample_count
FROM sensor_hourly
GROUP BY time_bucket('1 day', bucket), sensor_id, warehouse_id, sensor_type;

SELECT add_continuous_aggregate_policy('sensor_daily',
    start_offset => INTERVAL '3 days',
    end_offset => INTERVAL '1 day',
    schedule_interval => INTERVAL '1 day'
);
```
