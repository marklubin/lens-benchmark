TOPIC 2: TEST STRATEGY (3:32 PM — 3:52 PM)

[Rita Espinoza — 3:32 PM]
Thanks for inviting me. I wanted to discuss our testing pyramid because I think it's inverted. Let me share some numbers.

I pulled these from the CI dashboard this morning, so they're current as of today.

Current test distribution:
- Unit tests: 247
- Integration tests: 412
- End-to-end (e2e) tests: 86

Total: 745 tests.

The test pyramid says we should have many unit tests, fewer integration tests, and fewest e2e tests. We have it backwards — more integration tests than unit tests.

Why this matters: our CI pipeline takes 4 minutes 50 seconds. Of that, unit tests take 30 seconds, integration tests take 2 minutes 40 seconds, and e2e tests take 1 minute 40 seconds. The integration tests are the bottleneck.

More importantly, integration tests are flaky. Of the 14 flaky tests we fixed this sprint, 12 were integration tests. They depend on database state, network calls, and timing — all sources of non-determinism.

[Priti Sharma — 3:35 PM]
That CI time is also per-push, right? If someone pushes 3 times during code review, that's 15 minutes of CI just for one PR.

[Rita Espinoza — 3:35 PM]
Correct. And we average about 4 pushes per PR across the team. So the effective CI cost per merged PR is closer to 20 minutes.

[Derek Lane — 3:36 PM]
This is a fair critique. We got into the habit of writing integration tests because they were easier to reason about — set up the whole system, hit the endpoint, check the response. But that laziness is now costing us in CI time and reliability.

I'll own that as tech lead. Early on, I didn't push back on the integration-heavy approach because getting tests written at all felt like a win. But now that we have good test coverage, the composition matters.

[Angela Fong — 3:37 PM]
The payment reconciliation matching engine is a good example. It has 4 unit tests and 18 integration tests. Most of the integration tests are testing pure business logic (matching algorithms) through the HTTP layer. Those should be unit tests — test the matching function directly, no HTTP, no database.

I actually looked at this during the tech debt work this sprint. The 18 integration tests exercise maybe 6 distinct code paths. The rest are redundant — testing the same matching logic with different input data but going through the full stack each time.

[Marcus Webb — 3:38 PM]
When I wrote those tests, I was thinking about what to test, not how to test it. The integration test was the path of least resistance because the test fixtures were already set up.

Honestly, extracting the matching logic into a standalone function that can be unit-tested would also improve the code design. Right now the matching logic is buried inside the HTTP handler, which is why you can only test it via HTTP.

[Rita Espinoza — 3:39 PM]
That's a common pattern. My proposal:

1. **Set a target test ratio:** For every new feature, aim for 5:2:1 ratio of unit:integration:e2e tests.
2. **Gradually refactor existing tests:** Each sprint, convert 10 integration tests that test pure logic into unit tests. In 5 sprints, we'd have a properly shaped pyramid.
3. **Add a CI step that reports test type distribution.** Not a gate — just visibility. Team reviews in retro.

[Derek Lane — 3:40 PM]
I want to emphasize — converting an integration test to a unit test isn't just deleting the integration test and writing a unit test. Often it requires refactoring the production code to make the logic unit-testable. That's the real investment. The test conversion is the forcing function for better code architecture.

[Priti Sharma — 3:41 PM]
The 5:2:1 ratio — is that a hard rule or a guideline?

[Rita Espinoza — 3:42 PM]
Guideline. Some features are inherently more integration-heavy (data pipeline connections, for example). The point is to default to unit tests and justify integration tests when needed.

In code review, the question should shift from "does this have tests?" to "are these the right type of tests?" If someone submits a PR with 0 unit tests and 8 integration tests, the reviewer should ask why.

[Yuto Hayashi — 3:43 PM]
I like the refactoring approach. Converting integration tests to unit tests also makes them faster and more reliable. It's a win-win.

The infrastructure monitoring service I built has the opposite problem, actually — it's almost all unit tests because the external dependencies are heavily mocked. But I've discovered that the mocks drift from reality. So maybe the 5:2:1 ratio also helps by ensuring we have some integration tests to validate our mocks.

[Simone Osei — 3:44 PM]
Quick logistical question — where do contract tests fit? We have a few Pact tests for the partner API that aren't unit, integration, or e2e.

[Rita Espinoza — 3:44 PM]
Good question. I'd count contract tests as a subset of integration tests for the ratio. They test the interaction between two services, even though the mechanism is different from a traditional integration test.

[5 minutes elapsed — vote to extend: 5/7 yes]

[Derek Lane — 3:45 PM]
One more thing — our e2e tests are valuable but they run in a Docker Compose environment that takes 45 seconds to spin up. Can we share that environment across test runs instead of creating/destroying it each time?

Right now every CI run does: docker-compose up, run tests, docker-compose down. If we kept the environment warm between runs and just reset the database state, we'd save most of that 45 seconds.

[Rita Espinoza — 3:46 PM]
Yes. I've been working on a persistent test environment that resets state between runs instead of rebuilding. Estimated savings: 30 seconds per e2e run.

The approach is to use a test database that gets truncated between runs rather than dropped and recreated. The Docker containers stay up. I have a prototype branch that I'll clean up and share this sprint.

[Marcus Webb — 3:47 PM]
Would that persistent environment be shared across PRs? Because if two PRs run e2e tests at the same time, they might step on each other.

[Rita Espinoza — 3:47 PM]
No, each CI runner gets its own environment. The persistence is per-runner, not global. So there's no contention between parallel PR builds.

[Kai Nakamura — 3:47 PM]
**Decisions:**
1. **We decided to adopt a 5:2:1 unit:integration:e2e test ratio guideline for new features.**
2. **We decided to allocate effort each sprint to convert integration tests to unit tests.** Target: 10 conversions per sprint.
3. **We decided to add a CI step reporting test type distribution.** Non-blocking, reviewed in retros.
4. Rita will implement persistent e2e test environment to reduce spin-up time.

[Rita Espinoza — 3:48 PM]
I'll have the persistent environment PR ready for review by end of next week. I'll tag Derek for the review since it touches CI configuration.

TOPIC 3: SPRINT LENGTH — 1-WEEK SPRINTS? (3:52 PM — 4:07 PM)

[Priti Sharma — 3:52 PM]
I wrote this topic. We've been doing 2-week sprints since the team was formed. I want to propose switching to 1-week sprints. Here's why:

1. **Faster feedback loops.** With 2-week sprints, we plan on Day 1 and get retro feedback on Day 10. That's 10 working days of potential drift before we course-correct. With 1-week sprints, the feedback loop is 5 days.

2. **Smaller batch sizes.** 1-week sprints naturally force smaller tickets. We'd be less likely to have 13-point monsters like the reconciliation retry logic. Smaller tickets are easier to estimate, review, and deploy.

3. **More frequent delivery.** Shipping every week keeps momentum high and gives stakeholders more frequent touchpoints.

I know this is a big change, so I'm not saying we have to decide today. But I want to get the team's read on it.

[Derek Lane — 3:55 PM]
Counter-argument: 1-week sprints double the ceremony overhead. We'd have planning, standup, review, and retro every single week instead of every other week. That's 3-4 hours per week in meetings instead of the current 3-4 hours per two weeks.

For a 7-person team, that's 21-28 person-hours of ceremony per week. Currently it's 21-28 person-hours per two weeks. That's a meaningful difference.

[Marcus Webb — 3:56 PM]
Also, some work genuinely takes more than a week. The reconciliation retry logic took 6 days of implementation after the design review. In a 1-week sprint, that's the entire sprint — no room for anything else.

And that's 6 days of focused work. If we add the ceremony overhead from a mid-work sprint boundary, it could stretch to 7 or 8 days because of context switching.

[Angela Fong — 3:57 PM]
We could do 1-week sprints with multi-sprint epics for larger work. The sprint commitment would be "make progress on the epic" rather than "complete the epic."

[Simone Osei — 3:58 PM]
That feels like we'd be doing kanban with weekly checkpoints, not really sprint planning.

[Yuto Hayashi — 3:58 PM]
Simone makes a good point. If we're committing to "make progress" instead of "complete this deliverable," we've lost the core value of sprint planning, which is predictable delivery of specific outcomes.

[Priti Sharma — 3:59 PM]
That might not be a bad thing? We're already doing async standups. Maybe we should consider kanban with weekly retros instead of sprints with biweekly retros.

[Marcus Webb — 4:00 PM]
I'd be careful about switching to kanban. We tried kanban on the Atlas team two years ago and it turned into "work on whatever feels important this week" with no prioritization discipline. Sprint commitments, for all their overhead, create accountability.

[Kai Nakamura — 4:00 PM]
OK, this is a bigger conversation. The question isn't just sprint length — it's our process model. Let me reframe: what problem are we solving?

[Priti Sharma — 4:01 PM]
The 2-week estimation is unreliable. We hit 100% last sprint but historically we average 85%. With 1-week sprints, each estimate covers less time, so the estimation error is smaller.

[Derek Lane — 4:02 PM]
I think the root cause isn't sprint length — it's estimation technique. We could improve estimation within 2-week sprints by breaking tickets down more aggressively. The design review policy is already helping with this.

Look at the data: Sprint 14 was 81%, Sprint 15 was 100%, Sprint 16 was 96%. The upward trend started when we introduced mandatory design reviews and the pairing policy. If we keep improving the process within 2-week sprints, we might solve the estimation problem without the ceremony overhead of weekly sprints.

[Priti Sharma — 4:03 PM]
That's fair, but the 81% in Sprint 14 was partly because we had a 13-point ticket that turned out to be much harder than estimated. If we'd broken it down, we might have caught that earlier.

[Derek Lane — 4:04 PM]
Which is exactly the argument for ticket splitting, not shorter sprints. If the 13-pointer had been three 4-5 point tickets, we would have finished two of them and carried one — still better velocity than what happened.

[5 minutes elapsed — vote to extend: 4/7, needs 5 — no extension]

[Kai Nakamura — 4:04 PM]
Not enough votes to continue. Let me propose a middle ground: **we decided to keep 2-week sprints but start breaking tickets larger than 5 points into sub-tickets.** This gives us better estimation granularity without doubling our ceremony overhead. We'll revisit the sprint length question in 3 sprints based on our velocity trend.

If our velocity consistently stays above 95% with the ticket-splitting approach, it means estimation accuracy is the issue and we've addressed it. If velocity stays below 90% despite ticket splitting, that's evidence that sprint length might actually be the problem, and we can revisit.

[Priti Sharma — 4:05 PM]
Fair. I'll accept the compromise. If velocity stabilizes above 95% with the ticket-splitting approach, I'll drop the 1-week sprint proposal.

[Derek Lane — 4:06 PM]
For practical implementation — ticket splitting should happen during sprint planning or during design review, whichever comes first. If someone brings a ticket to planning that's over 5 points, we split it on the spot before committing.

[Kai Nakamura — 4:06 PM]
Agreed. I'll add that as a planning checklist item.

[Yuto Hayashi — 4:07 PM]

[Derek Lane — 4:09 PM]

[Simone Osei — 4:10 PM]

The format was roughly: 20 minutes of architecture overview with diagrams, 20 minutes of live code walkthrough, 20 minutes of Q&A. The Q&A was usually the most valuable part because people asked questions the presenter hadn't thought to cover.

[Angela Fong — 4:12 PM]
