[Mike Torres]
- Spent Friday debugging the flaky integration tests in the billing module. Turned out to be a race condition in the Stripe webhook handler — we were processing idempotency keys before the transaction committed. The flow was: webhook comes in, we start a DB transaction, check the idempotency key, process the event, then commit. But if a retry came in before the first transaction committed, the idempotency check would miss it because the key wasn't visible yet (read committed isolation). Fixed by adding a `SELECT ... - Today: picking up FALCON-12, the rate limiter refactor. Going to move the sliding window logic from the application layer into a Lua script on Redis so we stop hitting the double-count bug. Right now the check-and-increment is two separate Redis commands, and between those two commands another request can sneak in and also pass the check. Classic TOCTOU. The Lua script makes it atomic — check the sorted set count, add the new entry, trim old entries, all in one EVALSHA call.
- For reference, the current approach in `rate_limiter.py` looks like:


  And I'm replacing it with a single `EVALSHA` that does both operations atomically.

[James Holden]
- Wrapped up the data retention policy implementation (FALCON-04). PR #1849 is up. It adds TTL-based cleanup for the raw events table — anything older than 90 days gets archived to cold storage, then purged after 180. The archival process works in batches of 10,000 rows, writes them as gzipped Parquet files to S3, transitions the objects to Glacier after 30 days via an S3 lifecycle policy, and then deletes the source rows from Postgres in a separate cleanup pass.
- The batch size of 10,000 was chosen based on testing — at that size, each batch takes about 1.2 seconds to read, serialize, and upload, and the DELETE doesn't hold a lock long enough to interfere with production reads. I tested with batch sizes of 1K, 5K, 10K, 25K, and 50K:

  | Batch Size | Read+Upload (s) | Delete Lock (ms) | Total Time (min) |
  |------------|-----------------|-------------------|-------------------|
  | 1,000      | 0.3             | 12                | 47                |
  | 5,000      | 0.7             | 45                | 14                |
  | 10,000     | 1.2             | 82                | 8.5               |
  | 25,000     | 2.8             | 210               | 5.2               |
  | 50,000     | 5.1             | 480               | 4.1               |

  The 25K and 50K sizes start to show noticeable lock contention during peak hours so 10K is the sweet spot.

- SUP-4421: Session lookup failed at 2025-05-29 11:47:23 UTC, session created at 2025-05-29 09:12:08 UTC (2h35m old, should still be valid — our session TTL is 24 hours)
- SUP-4423: Session lookup failed at 2025-05-30 14:02:51 UTC, session created at 2025-05-30 10:30:17 UTC (3h32m old)

[Priya Chandrasekaran]
That's the third time this month we've heard about random logouts. There was that batch of tickets two weeks ago too — SUP-4398 through SUP-4402. Five users in one day.

[Mike Torres]
Yeah, I looked into those earlier ones. The session store is PostgreSQL-backed right now — we're using the `django.contrib.sessions` database backend. The relevant settings in our `settings.py` are:

```python
SESSION_ENGINE = "django.contrib.sessions.backends.db"
SESSION_COOKIE_AGE = 86400  # 24 hours
SESSION_SAVE_EVERY_REQUEST = True
SESSION_COOKIE_SECURE = True
SESSION_COOKIE_HTTPONLY = True
```

The problem is twofold: first, we're running session cleanup as a cron job that deletes expired sessions, but the expiry check has a timezone bug. It's comparing UTC timestamps against session creation times that were stored in the server's local timezone. The cron job runs `DELETE FROM django_session WHERE expire_date < NOW()`, but `expire_date` was being set using `datetime.now()` instead of `datetime.utcnow()` in one of our custom session middleware patches — and our servers are running in US-Pacific, which is UTC-7. So sessions created during certain hours look like they expired 7 hours earlier than they actually did. Any session created between 00:00 and 07:00 UTC (5pm to midnight Pacific) gets an `expire_date` that, when compared to UTC `NOW()`, appears to already be expired.

Second issue is that when the database connection pool is saturated — which happens during our traffic spikes between 10am and 2pm ET — session lookups time out and Django falls back to creating a new session, which invalidates the old one from the user's perspective. The connection pool is configured with `CONN_MAX_AGE = 600` and `MAX_CONNS = 20` via pgBouncer, but the session lookup, the user lookup, and the actual request handler are all competing for the same pool.

[Sarah Kim]
That connection pool saturation is going to get worse as we scale. We're at what, 200 concurrent users during peak now? The session table has 14,000 rows and growing.

[Mike Torres]
15,200 as of this morning. And the session lookup is hitting the table on every authenticated request. Even with the index, it's adding 2-3ms per request at current load, which is fine, but at 500 concurrent users it'll start competing with the actual application queries for pool connections. I ran a quick simulation by replaying our production query log at 2.5x speed on staging, and at that load the p99 for session lookups went from 3ms to 47ms, and the application query p99 went from 18ms to 89ms. The pool was at 100% utilization 60% of the time during that test.

[Ben Nakamura]
Is there a way to reproduce this in our test suite? If we're going to fix this, I want regression tests that would catch if we introduce a similar scaling issue in the future.

[Mike Torres]
Good question. We could add a load test to the CI pipeline that runs against a test database with a realistic number of sessions. But it'd be slow — probably 2-3 minutes even with a stripped-down dataset. Might be better as a nightly job rather than per-PR.

[Priya Chandrasekaran]
This sounds like it's becoming a real problem. Auth is becoming a problem. Users getting logged out randomly — that's not acceptable for a B2B product where people are in the middle of building dashboards. We need to rethink this.

[David Okafor]
Agreed. The Postgres session backend was fine when we had 30 users but it's clearly not scaling. I see the same pattern with the events table query I'm working on — Postgres is great for relational data but when you use it as a key-value store for high-frequency lookups, you're fighting against its architecture.

[Mike Torres]
I've been thinking about this. The fundamental issue is that we're using a general-purpose relational database as a session store. Sessions are essentially a key-value lookup pattern — you have a session ID and you need to get the associated data as fast as possible. PostgreSQL can do it but it's not optimized for it. The overhead of MVCC, WAL writes, vacuum, index maintenance — none of that adds value for a workload that's just "get value by key, set value by key, delete key after TTL."
