ITEM 2: PRICING PAGE REDESIGN DECISION (11:24 AM–11:46 AM)

[Claire Dunning — 11:24 AM]
We've been iterating on the pricing page redesign for three sprints now. I want to lock in a direction today. Let me present the final two options.

[Claire Dunning — 11:25 AM]
Background: our current pricing page has three tiers (Free, Pro, Enterprise) displayed as static cards. Conversion rate from pricing page to signup is 8.2%. Industry benchmark for B2B SaaS is 12-15%. We're underperforming.

User research (24 interviews) surfaced three issues:
1. Feature comparison is confusing — users can't quickly tell the difference between Pro and Enterprise
2. The Enterprise tier has no price shown ("Contact Sales") which makes users assume it's unaffordable
3. The Free tier limitations aren't clear — users sign up for Free and hit limits within days, leading to frustration

[Diana Park — 11:27 AM]
I want to underscore finding #2 from the user research. In 16 of the 24 interviews, users said they assumed Enterprise was "probably $2,000 a month or something crazy" because there was no price shown. The reality is most Enterprise deals close between $499 and $1,200 per month. The perception gap is massive. Multiple users said they didn't even consider Enterprise because of the "Contact Sales" barrier.

[Claire Dunning — 11:28 AM]
Thanks Diana. So here are the two options we've narrowed down to:

**Option A: Interactive pricing calculator**
A dynamic pricing page where users select their team size, expected data volume, and feature requirements. The page calculates a recommended plan and shows a price estimate, even for Enterprise (based on the selections). Users can toggle features to see how the price changes.

Pros: Transparent, engaging, addresses all three user research findings.
Cons: Complex to build (2-3 sprint effort), maintenance burden (pricing changes require calculator updates), might anchor users to lower prices during sales calls.

**Option B: Comparison table with quick-start CTAs**
A clean comparison table with checkmarks/X marks for each feature across tiers. Each tier has a prominent CTA button. Enterprise shows a starting price ("From $499/mo") instead of "Contact Sales." Each tier card has a "Quick start" button that goes directly to signup with that plan pre-selected.

Pros: Simple, fast to build (1 sprint), addresses the comparison confusion and Enterprise price transparency issues.
Cons: Less engaging than the calculator, doesn't personalize the recommendation.

[Claire Dunning — 11:31 AM]
Design prototypes for both are in the Figma board (link in Slack). User testing results:

| Metric | Current | Option A | Option B |
|--------|---------|----------|----------|
| Time to choose a plan | 3.2 min | 1.8 min | 1.4 min |
| Confidence in choice | 3.1/5 | 4.3/5 | 4.0/5 |
| Perceived value | 3.4/5 | 4.5/5 | 4.1/5 |
| "Would sign up" | 42% | 71% | 64% |

Option A scores higher across the board, but Option B is competitive and much simpler to build.

ITEM 3: DASHBOARD FEATURE GATING (11:46 AM–12:02 PM)

[Brooke Tamura — 11:46 AM]
Quick topic. We need to decide on the feature gating strategy for the new dashboard capabilities (custom layouts, saved views, scheduled reports). The question is: which features are Free, which are Pro, which are Enterprise?

[Brooke Tamura — 11:47 AM]
Current state: all dashboard features are available to all tiers. No gating. This was fine when dashboards were basic, but the new capabilities represent significant value and we need to monetize them.

I've gone through three rounds of iteration on this with input from sales, support, and engineering. The v3 proposal is in the pre-read doc, but let me walk through it here.

My proposal:

| Feature | Free | Pro | Enterprise |
|---------|------|-----|------------|
| Basic dashboards (pre-built layouts) | Yes | Yes | Yes |
| Custom layouts (drag-and-drop) | No | Yes | Yes |
| Saved views | 3 max | Unlimited | Unlimited |
| Scheduled reports | No | 5/month | Unlimited |
| Dashboard sharing (public link) | No | Yes | Yes |
| Dashboard embedding (iframe) | No | No | Yes |
| White-label embedding | No | No | Yes |
| Real-time updates (WebSocket) | No | Yes | Yes |

[Brooke Tamura — 11:50 AM]
The key tradeoff: we want Free tier to be useful enough to convert users to Pro, but limited enough to incentivize the upgrade.

[Brooke Tamura — 11:50 AM]
I also want to flag the implementation approach. Jason and I discussed this last week. The gating will use our existing feature flag system — we'll tie each capability to a plan-level flag. The backend enforcement is straightforward. The UX question is how we handle the locked features — do we hide them entirely, or show them as disabled with an upgrade prompt?

[Naveen Gupta — 11:51 AM]
Show them as disabled with an upgrade prompt. Always. Hiding features means users don't know what they're missing. A grayed-out option with a "Pro" badge and a tooltip that says "Upgrade to Pro to unlock custom layouts" is much more effective at driving upgrades than just not showing the option.
