[Leah Kim — 2:08 PM]
Sure. My recommendation is **PostgreSQL 16**.

I sent out a detailed write-up last night, but I'll summarize the key points here.

Rationale:

1. **Relational model fit.** The inventory data model is inherently relational — products, categories, warehouses, stock levels, reservations, transfers. PostgreSQL's relational model maps directly to the domain. No impedance mismatch. We're not trying to squeeze a document model into relational tables or vice versa. The entities and relationships are clean.

2. **ACID compliance.** Reservations require strong consistency. When a customer reserves 5 units, that stock must be immediately unavailable to other orders. PostgreSQL's transaction isolation (Serializable or Repeatable Read) guarantees this without application-level locking. We can also use advisory locks if we need finer-grained control, but for the reservation use case, standard row-level locking via SELECT FOR UPDATE is sufficient and well-understood.

3. **Write performance.** PostgreSQL 16 can sustain 50K TPS on commodity hardware with proper tuning (pg_tune, shared_buffers, WAL configuration). Our requirement is 300 TPS (18K/min). We're at 0.6% of PostgreSQL's capacity. Plenty of headroom. I ran pgbench with a custom workload that simulates our reservation flow — on a db.r6g.xlarge (which is what Fatima would provision), I got 4,200 TPS sustained. That's 14x our peak requirement.

4. **Operational maturity.** We already run PostgreSQL 15 in production for three other services — AuthService, NotificationHub, and the billing ledger. The team has deep expertise. Fatima has the monitoring, backup, and failover infrastructure already built. We're not starting from zero.

5. **Ecosystem.** pg_cron for scheduled reporting queries, pg_partman for table partitioning (we'll want to partition stock_level_history by month), pgvector if we ever do similarity search on product descriptions. There's also excellent tooling — pgAdmin, pg_stat_statements for query analysis, pg_repack for online table reorganization.

6. **Cost.** Open source. Our existing RDS infrastructure supports it. No new licensing costs. And Greg, to your point — the official Docker image is 400MB, starts in 2 seconds, and our CI already has it.

The one concern I'll flag proactively: PostgreSQL's MVCC (Multi-Version Concurrency Control) can lead to table bloat on high-write tables. The stock_levels table will get 18K updates/minute. We'd need autovacuum aggressively tuned, and potentially partition the table by warehouse to limit vacuum scope. I've already drafted the partitioning strategy.

```sql
-- Proposed stock_levels schema
CREATE TABLE stock_levels (
    warehouse_id UUID NOT NULL,
    product_id UUID NOT NULL,
    available_qty INT NOT NULL DEFAULT 0,
    reserved_qty INT NOT NULL DEFAULT 0,
    in_transit_qty INT NOT NULL DEFAULT 0,
    last_updated TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    version INT NOT NULL DEFAULT 1,  -- optimistic locking
    PRIMARY KEY (warehouse_id, product_id)
) PARTITION BY HASH (warehouse_id);

-- 16 partitions (one per hash bucket, roughly 21 warehouses each)
CREATE TABLE stock_levels_p0 PARTITION OF stock_levels FOR VALUES WITH (MODULUS 16, REMAINDER 0);
CREATE TABLE stock_levels_p1 PARTITION OF stock_levels FOR VALUES WITH (MODULUS 16, REMAINDER 1);
-- ... etc
```

The hash partitioning distributes writes evenly across partitions, which means autovacuum can process each partition independently. This prevents the "vacuum can't keep up" problem that plagues large un-partitioned tables. I've seen this exact approach work at my previous company for a table doing 25K updates/minute — vacuum never fell behind once we moved from a single table to 16 hash partitions.

[Omar Zafar — 2:16 PM]
Quick clarification — why hash on warehouse_id rather than product_id?

[Leah Kim — 2:16 PM]
Two reasons. First, most of our queries are warehouse-scoped — "show me all stock for warehouse X." Hash on warehouse_id means those queries hit a single partition. Second, warehouse_id has better cardinality distribution for our workload. The top 20 warehouses generate 60% of the transactions, so we get natural load balancing across partitions. If we hashed on product_id, the long-tail products would create uneven partition sizes.

[Ben Torres — 2:18 PM]
I also evaluated PostgreSQL — and I agree with everything Leah said about it. It's the safe, proven choice. But I want to present an alternative that I think deserves serious consideration: **CockroachDB**.

CockroachDB gives us everything PostgreSQL gives us (SQL, ACID, relational model) plus:

1. **Distributed by default.** CockroachDB is a distributed SQL database. Data is automatically replicated across 3+ nodes. If a node fails, the cluster continues serving reads and writes. No failover configuration, no replica promotion — it's built in. With PostgreSQL on Aurora, we have a primary and read replicas. If the primary fails, Aurora promotes a replica — that's a 30–60 second failover window. CockroachDB's consensus-based approach has zero failover time because there's no single primary.

2. **Horizontal write scaling.** With PostgreSQL, writes go to the primary. To scale writes, you scale up (bigger instance). With CockroachDB, writes are distributed across nodes. To scale, you add nodes. Our 18K writes/minute today might be 180K in 3 years — CockroachDB scales linearly. PostgreSQL scaling would mean progressively larger and more expensive instances.

3. **Multi-region capability.** If we expand to international warehouses (the 2026 roadmap mentions EMEA expansion), CockroachDB can replicate data to EU nodes with follower reads for low-latency access. PostgreSQL would need a complex replication setup. CockroachDB's locality-aware routing means EU queries read from EU nodes automatically.

4. **Built-in change data capture (CDC).** CockroachDB's changefeed feature streams row-level changes to Kafka. This is perfect for the analytics pipeline — Nadia can consume stock level changes in real-time without polling. You set up a changefeed with a single SQL statement and it just works.

The trade-offs, and I want to be upfront about these:
- **Cost.** CockroachDB Dedicated (managed cloud) is ~3x the cost of RDS PostgreSQL for equivalent specs. I got a rough quote from their sales team: $5,400/month for a 3-node cluster with our specs versus $1,840 for the Aurora setup Fatima specced.
- **Latency.** CockroachDB's distributed consensus (Raft) adds ~5ms to each write compared to PostgreSQL. Our P99 requirement is 50ms for reads, so this is fine for writes, but reads through distributed consensus are slower than PostgreSQL's local reads.
- **Operational unfamiliarity.** Nobody on the team has run CockroachDB in production. The debugging story is different — different monitoring tools, different query plan analysis, different operational runbooks.

Here are benchmark numbers I ran on a 3-node CockroachDB cluster (8 vCPUs, 32GB RAM per node). I used a workload modeled on our reservation flow — mixed reads and writes with the same cardinality as our production data:

| Operation | PostgreSQL 16 (single) | CockroachDB (3-node) |
|-----------|------------------------|----------------------|
| INSERT (stock update) | P50: 1.2ms, P99: 4.1ms | P50: 6.8ms, P99: 18ms |
| SELECT (stock lookup) | P50: 0.4ms, P99: 1.8ms | P50: 2.1ms, P99: 8.4ms |
| Transaction (reserve + update) | P50: 3.1ms, P99: 12ms | P50: 14ms, P99: 38ms |

PostgreSQL is faster in every category because it's a single-node system — no network round-trips for consensus. But CockroachDB is well within our latency requirements. The P99 for the full reservation transaction is 38ms on CockroachDB, and our budget is 50ms. That's a 12ms margin. Not huge, but sufficient.

SCHEMA DESIGN DISCUSSION (2:55 PM — 3:25 PM)

[Leah Kim — 2:55 PM]
Alright, for those of you still here. While we're all together, let me walk through the proposed schema. I want to validate the data model before I start coding the migration scripts this weekend.

I'll go table by table. Stop me if anything looks wrong or if I'm missing a use case.

**Core tables:**

```sql
-- Product catalog (2.4M rows, read-heavy)
CREATE TABLE products (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    sku VARCHAR(50) UNIQUE NOT NULL,
    name TEXT NOT NULL,
    category_id UUID REFERENCES categories(id),
    unit_of_measure VARCHAR(20) NOT NULL,
    weight_grams INT,
    dimensions_cm JSONB,  -- {length, width, height}
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Warehouse locations (340 rows, rarely changes)
CREATE TABLE warehouses (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    code VARCHAR(10) UNIQUE NOT NULL,
    name TEXT NOT NULL,
    region VARCHAR(50) NOT NULL,
    address JSONB NOT NULL,
    capacity_cubic_meters INT,
    is_active BOOLEAN DEFAULT true
);

-- Stock levels (2.4M × 340 = ~816M potential rows, but sparse — avg 12 warehouses per product)
-- Actual rows: ~28.8M
-- Write-heavy: 18K updates/minute
CREATE TABLE stock_levels (
    warehouse_id UUID NOT NULL REFERENCES warehouses(id),
    product_id UUID NOT NULL REFERENCES products(id),
    available_qty INT NOT NULL DEFAULT 0 CHECK (available_qty >= 0),
    reserved_qty INT NOT NULL DEFAULT 0 CHECK (reserved_qty >= 0),
    in_transit_qty INT NOT NULL DEFAULT 0 CHECK (in_transit_qty >= 0),
    last_transaction_id UUID,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    version INT NOT NULL DEFAULT 1,
    PRIMARY KEY (warehouse_id, product_id)
) PARTITION BY HASH (warehouse_id);

-- Reservations (ACID-critical, moderate write rate)
CREATE TABLE reservations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id UUID NOT NULL,
    warehouse_id UUID NOT NULL,
    product_id UUID NOT NULL,
    quantity INT NOT NULL CHECK (quantity > 0),
    status VARCHAR(20) NOT NULL DEFAULT 'active',  -- active, fulfilled, cancelled, expired
    expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    FOREIGN KEY (warehouse_id, product_id) REFERENCES stock_levels(warehouse_id, product_id)
);

-- Stock transactions (append-only audit trail)
CREATE TABLE stock_transactions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    warehouse_id UUID NOT NULL,
    product_id UUID NOT NULL,
    transaction_type VARCHAR(30) NOT NULL,  -- inbound, outbound, transfer_in, transfer_out, adjustment, reservation, release
    quantity INT NOT NULL,
    reference_id UUID,  -- order_id, transfer_id, or adjustment_id
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by UUID NOT NULL
) PARTITION BY RANGE (created_at);
```

[Jake Petrov — 3:05 PM]
The reservation flow — walk me through the transaction. Customer places an order, we need to reserve stock. What's the SQL?

[Leah Kim — 3:06 PM]
Sure. Here's the full reservation flow as a single transaction:

```sql
BEGIN;

-- Lock the stock row (SELECT FOR UPDATE prevents concurrent modifications)
SELECT available_qty, version
FROM stock_levels
WHERE warehouse_id = $1 AND product_id = $2
FOR UPDATE;

-- Check availability (application logic)
-- If available_qty < requested_qty: ROLLBACK

-- Reserve stock
UPDATE stock_levels
SET available_qty = available_qty - $3,
    reserved_qty = reserved_qty + $3,
    version = version + 1,
    updated_at = NOW()
WHERE warehouse_id = $1 AND product_id = $2 AND version = $4;

-- Create reservation record
INSERT INTO reservations (order_id, warehouse_id, product_id, quantity, expires_at)
VALUES ($5, $1, $2, $3, NOW() + INTERVAL '30 minutes');

-- Create audit transaction
INSERT INTO stock_transactions (warehouse_id, product_id, transaction_type, quantity, reference_id, created_by)
VALUES ($1, $2, 'reservation', $3, $5, $6);

COMMIT;
```

The `SELECT FOR UPDATE` locks the specific stock row, preventing other transactions from modifying it until we commit. The `version` check provides optimistic locking as a safety net — if the version doesn't match, someone else modified the row between our SELECT and UPDATE (shouldn't happen with `FOR UPDATE`, but defense in depth). If the UPDATE affects 0 rows because the version changed, the application retries the whole transaction.

[Jake Petrov — 3:09 PM]
What's the lock scope? Does the `FOR UPDATE` lock only the one row, or the whole partition?

[Leah Kim — 3:09 PM]
Just the one row. PostgreSQL's row-level locking is granular — `FOR UPDATE` acquires a row-level exclusive lock on the matching rows only. Other transactions can read and write other rows in the same partition concurrently. The only contention is when two transactions try to reserve the same product in the same warehouse simultaneously, which is the exact case where we need serialization.

[Ben Torres — 3:10 PM]
What about reservation expiry? If a customer doesn't complete their order in 30 minutes, the reserved stock needs to be released. We can't have phantom reservations eating up available inventory.

[Leah Kim — 3:11 PM]
Two approaches. Option A: background job that runs every minute, finds expired reservations, releases stock. Option B: use pg_cron to run a stored procedure.

I prefer Option A because the business logic (release stock, update reservation status, create audit transaction) belongs in the application layer, not in a database procedure. Stored procedures are harder to test, harder to version control, and harder to debug in production. The job is simple and runs in the application's existing task scheduler:

```python
async def release_expired_reservations():
    expired = await Reservation.filter(
        status='active',
        expires_at__lt=now()
    ).select_for_update()

    for reservation in expired:
        async with transaction.atomic():
            await StockLevel.filter(
                warehouse_id=reservation.warehouse_id,
                product_id=reservation.product_id
            ).update(
                available_qty=F('available_qty') + reservation.quantity,
                reserved_qty=F('reserved_qty') - reservation.quantity,
                version=F('version') + 1
            )
            reservation.status = 'expired'
            await reservation.save()
            await StockTransaction.create(
                warehouse_id=reservation.warehouse_id,
                product_id=reservation.product_id,
                transaction_type='release',
                quantity=reservation.quantity,
                reference_id=reservation.order_id,
                created_by=SYSTEM_USER_ID
            )
```

[Ben Torres — 3:14 PM]
Looks clean. One edge case — what if the job is running and a customer completes their order at the same time? The `select_for_update` should prevent a race condition, but we should test it.

[Leah Kim — 3:14 PM]
Right. The `select_for_update` on the reservation row means the job can't process a reservation that the order-fulfillment path is simultaneously trying to fulfill. One of them will wait for the other's transaction to commit. If the fulfillment path commits first, the reservation status will be 'fulfilled', and the job's query won't match it. If the job commits first, the fulfillment path will see status='expired' and return an error to the customer, who would need to re-reserve. That's the correct behavior.
