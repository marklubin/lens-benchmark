ITEM 1: ADR-017 COUNTER-PROPOSAL — JWT REVOCATION ANALYSIS (2:00–2:30 PM)

[Priya Chandrasekaran — 2:01 PM]
OK, picking up where we left off Tuesday. Mike, you had time to think through an alternative. The floor is yours.

[Mike Torres — 2:02 PM]
Thanks. So I want to start by saying Sarah's JWT analysis was thorough and the stateless model is appealing. My issue isn't with JWT in general — it's with JWT for THIS use case, where we have enterprise customers who require immediate session revocation.

Let me frame the core tension. When we say we want to move off the PostgreSQL session store, what we really want is:
1. Fast session lookups that don't compete with application queries for database connections
2. Horizontal scalability — any API server can validate any session
3. Reliable session lifecycle management — no more timezone bugs in cleanup
4. Immediate revocation capability for enterprise compliance

JWT nails requirements 1-3 but struggles with requirement 4. That's the tension. If we implement a token blacklist for revocation, we reintroduce server-side state, and at that point we're maintaining both a stateless token system AND a stateful revocation store. Two systems instead of one.

So my question was: is there a single system that satisfies all four requirements? And the answer is yes — but it requires letting go of the "stateless" paradigm.

Before I get to my proposal, let me share some data I pulled. Let me share my screen.

[Mike shares screen — terminal output and Grafana dashboard]

[Lena Novak — 2:05 PM]
Can you zoom in a bit? The Grafana panel text is small on my end.

[Mike Torres — 2:05 PM]
Sure, let me bump the browser zoom. Better?

[Lena Novak — 2:05 PM]
Perfect, thanks.

[Mike Torres — 2:06 PM]
OK, so this is the session-related query performance panel. I pulled this from our production Grafana. The time range is the last 7 days, broken out by percentile.

Here's the session lookup latency distribution for the last 7 days:
- P50: 2.1ms
- P95: 8.4ms
- P99: 47ms
- P99.9: 312ms

The P99 and P99.9 numbers are the problem — those are the requests hitting during connection pool saturation. But look at P50 — 2.1ms for a PostgreSQL lookup with an indexed query is actually fine. The issue isn't that PostgreSQL is slow; it's that PostgreSQL is shared between session lookups and everything else.

Let me switch to the connection pool dashboard. Here are the connection pool stats during peak (10am-2pm ET):
- Pool size: 20 connections
- Active connections: 18-20 (saturated)
- Waiting queue: 0-12 (this is where latency spikes come from)
- Session lookup queries: 34% of all queries during peak

[Carlos Reyes — 2:08 PM]
Quick clarification — the 34% figure, is that 34% of all queries or 34% of all connection time?

[Mike Torres — 2:08 PM]
34% of query count, not connection time. Session lookups are fast queries — they're usually under 2ms each — so they're not holding connections long. But they're frequent enough that they contribute meaningfully to pool contention. Each API request starts with a session lookup, so the session query count roughly equals total authenticated request count.

[Carlos Reyes — 2:09 PM]
Got it. So the issue is that each of those lookups needs a connection from the pool, even if it only holds it for a couple milliseconds.

[Mike Torres — 2:09 PM]
Exactly. It's connection acquisition time, not query execution time. When the pool is saturated, that lookup that takes 2ms to execute might wait 45ms for a connection. That's where the P99 spike comes from.

So session lookups are consuming 34% of our database connection pool during peak hours. That's the root cause. If we move sessions to a dedicated store — any dedicated store — we free up 34% of our connection pool AND eliminate the tail latency problem.

[Sarah Kim — 2:10 PM]
That's a good analysis. But if the argument is "move sessions to a dedicated store," JWT also achieves that — the store is the token itself.

[Mike Torres — 2:11 PM]
True, but JWT also introduces the revocation problem, which we've discussed. Let me present what I'm thinking and then we can compare.

The core of my concern is this: JWT's stateless model means the server has no record of active sessions. If a token is issued, the server cannot un-issue it. The token exists until it expires. This is a fundamental property of the architecture, not a bug to be worked around.

The revocation blacklist that Sarah mentioned — let me walk through what that actually looks like in practice.

Say we implement JWT with 15-minute access tokens and a blacklist for revocation. Here's the flow when an admin needs to revoke a user's sessions:

1. Admin calls POST /api/admin/users/{id}/revoke-sessions
2. Server generates the list of active JTIs (token IDs) for that user
3. Wait — how does the server know which JTIs are active? It doesn't. JWT is stateless. The server didn't record them.
4. OK, so we'd need to record JTIs at issuance. Now we have a JTI store — which is... a session store.
5. Alternatively, we maintain a "revoked users" list. On every request, after validating the JWT signature, we also check if the user ID is in the revoked list.
6. Now every request does: verify JWT signature + check revocation list. Two operations instead of one.
7. The revocation list needs to be fast — sub-millisecond lookups. So we'd put it in... Redis or an in-memory cache.

So the revocation path for JWT leads us to: JWT verification + Redis lookup on every request. We've added complexity (two auth systems) without eliminating the server-side lookup.

[Lena Novak — 2:15 PM]
From the frontend perspective, the dual-system thing also means we'd need to handle two different kinds of auth errors. If the JWT itself is expired, that's a 401 with a different recovery path than if the user is on the revocation list. We'd need to distinguish between "your token expired, refresh it" and "your session has been revoked, go back to login."

[Mike Torres — 2:16 PM]
That's a good point, Lena. The error semantics get muddier with two systems. With a single session store, there's one reason your session is invalid: it doesn't exist. Simple.

[David Okafor — 2:17 PM]
That's a fair characterization. When I worked at DataScale, we went down this exact path. Started with JWT, added a Redis blacklist for revocation, and eventually realized we were maintaining two auth systems that had to agree with each other. When they disagreed — and they did, during a Redis failover — we had a 20-minute window where revoked tokens were accepted because the blacklist wasn't available.

[Mike Torres — 2:18 PM]
Exactly. The failure mode of JWT-with-blacklist is that revocation silently stops working when the blacklist is unavailable. That's the opposite of what an enterprise customer wants.

[Carlos Reyes — 2:18 PM]
Just to underscore that from a compliance perspective — if we tell customers in our security questionnaire that we support immediate revocation, and our implementation has a failure mode where revocation silently degrades, that's a significant audit finding. The SOC 2 auditors would flag that. We'd need to document the failure mode and the mitigation, and "we hope Redis doesn't go down" is not a mitigation they'll accept.

[Sarah Kim — 2:19 PM]
OK, I hear you. So what's your alternative?

[Mike Torres — 2:20 PM]
I don't have a fully fleshed proposal yet — I wanted to present the JWT revocation analysis first because I think it changes the framing. If we accept that we need server-side state for sessions regardless, then the question becomes: what's the best server-side session store? Not "should we go stateless?"

The problems with PostgreSQL are:
- Shared connection pool with application queries
- Row-based storage is overkill for key-value lookup pattern
- Session cleanup requires a background job

What we want is a dedicated, fast, key-value store with built-in TTL support so sessions auto-expire without a cleanup job.

[James Holden — 2:21 PM]
Can I ask a quick scoping question? When we say "session," what's the payload? Is it literally just a session ID mapping to a user ID, or are we storing additional session state — preferences, feature flags, cart contents, anything like that?

[Mike Torres — 2:21 PM]
Good question. Right now, the Django session stores: user ID, CSRF token, last activity timestamp, and a few feature flag overrides. Total payload is around 400-600 bytes per session. It's small. We're not storing anything heavy in the session itself — the heavy data is in the user profile and tenant tables.

[James Holden — 2:22 PM]
OK, so it's truly a key-value pattern. That narrows the technology choices a bit — we don't need document storage or query capabilities.

[Sarah Kim — 2:22 PM]
Are you talking about Redis? Because if we put sessions in Redis, we still need to handle Redis availability, replication, failover...

[Mike Torres — 2:23 PM]
I'm not proposing a specific technology yet. I want to establish the requirements first. The requirements are:
1. Key-value lookup in sub-millisecond time
2. Built-in TTL for automatic session expiry
3. Not sharing resources with the application database
4. Native deletion for immediate revocation — just delete the key
5. Horizontal read scalability
6. Persistence across restarts (we can't lose all sessions on a deploy)

Redis fits most of these. Memcached fits some but not persistence. DynamoDB fits all but has higher latency and cost. There are tradeoffs.

But here's the key insight: with a dedicated session store that has built-in TTL, requirements 1-4 are satisfied by the store itself. No blacklists, no cleanup jobs, no dual-system complexity. Revocation is just `DELETE session_key` — the simplest operation possible.

[Rachel Fernandez — 2:25 PM]
Is persistence across restarts (requirement 6) a hard requirement or a nice-to-have? If we lost all sessions on a deploy, users would just have to re-login. That's annoying but not catastrophic.

[Mike Torres — 2:25 PM]
It's soft. We could live without it. But for enterprise customers, forcing 500 users to re-login simultaneously because of a deployment is a support ticket generator. It's nice to avoid.

[Priya Chandrasekaran — 2:26 PM]
OK, so Tuesday we heard the case for JWT. Today we've heard the case against JWT and a framing for a server-side session store. I think we need a concrete counter-proposal to compare against. Mike, can you write up a specific proposal in the ADR? Technology choice, implementation plan, migration path?

[Mike Torres — 2:27 PM]
Yeah, I'll update ADR-017 with the details. I want to benchmark a couple options first — I'll have something by Monday.

Alright. Let's plan to make a decision next week. We can't let this drag — the support tickets keep coming in. In the meantime, Mike, can you fix the timezone bug in the cleanup job? That's the immediate fire.

[Mike Torres — 2:28 PM]
Already have a fix locally — just need to write the test. I'll get a PR up today. It's a one-line change — the cleanup query needs to cast both timestamps to UTC before comparing.

Can I see the fix before it merges? The session cleanup touches auth security surface and I want to make sure the fix doesn't have any edge cases around DST transitions. We've had issues before with pytz and DST boundaries.

GITHUB — PR #1858: Fix timezone comparison in session cleanup job
Author: Mike Torres
Branch: fix/session-cleanup-tz
Status: Open, 1 approval (David Okafor), CI passing
Labels: bug, auth, priority:high
Linked issue: RIVULET-2847

**Description:**
The `clearsessions` management command compares `expire_date` (stored as UTC) against `timezone.now()` which returns a timezone-aware datetime in the server's local timezone (America/Los_Angeles). When the server time is UTC-7, sessions created between midnight and 7am UTC appear to be 7 hours older than they are, causing premature deletion.

**Root cause analysis:**
Django's `timezone.now()` returns the current time in the `TIME_ZONE` setting's timezone when `USE_TZ=True`. Our production servers are configured with `TIME_ZONE = 'America/Los_Angeles'`. The session `expire_date` is stored in UTC (PostgreSQL `timestamptz`). When Django compares these two timezone-aware datetimes, it correctly handles the conversion — but our custom cleanup query was constructing a raw SQL comparison without timezone conversion.

The relevant code path:
```python
# rivulet/auth/management/commands/clearsessions.py
class Command(BaseCommand):
    def handle(self, **options):
        # This was the buggy line — timezone.now() returns America/Los_Angeles
        Session.objects.filter(expire_date__lt=timezone.now()).delete()
```

**Fix:**
Force both sides of the comparison to UTC:

```python
# Before (buggy)
Session.objects.filter(expire_date__lt=timezone.now()).delete()

# After (correct)
from django.utils import timezone
import pytz
utc_now = datetime.now(pytz.UTC)
Session.objects.filter(expire_date__lt=utc_now).delete()
```

**Impact:**
- Affected window: sessions expiring between 00:00–07:00 UTC (5 PM–midnight PT)
- Symptom: users logged out unexpectedly during evening PT hours

[Mike Torres]
Hey, random thought — you mentioned you dealt with JWT + Redis blacklist at DataScale. What was the final architecture you ended up with there?

[David Okafor]
We eventually ripped out JWT entirely and went with Redis-backed sessions. Similar to what you're alluding to. The JWT blacklist was a maintenance nightmare — we had a bug where the blacklist TTL didn't match the token TTL, so revoked tokens would fall out of the blacklist before they expired. 3-month security incident.

[Mike Torres]
That's exactly the kind of failure mode I'm worried about. Two systems that need to agree on TTLs. How was Redis session performance at DataScale?

[David Okafor]
Excellent. Sub-millisecond lookups. We were doing 50K requests/sec at peak and Redis wasn't even close to breaking a sweat. The real benefit was operational simplicity — revocation was just DELETE, cleanup was built-in via EXPIRE, and failover was handled by Redis Sentinel. No custom code for any of it.

[Mike Torres]
What Redis deployment topology did you use? Single primary with read replicas, or Sentinel, or Cluster?

[David Okafor]
Sentinel with one primary and two replicas. For sessions, you don't need Cluster — the data volume is tiny. Even at 100K concurrent sessions at 500 bytes each, that's only 50MB. Redis handles that on a micro instance. Sentinel gives you automatic failover, which was the main concern. Failover typically completed in 10-15 seconds, during which reads still worked from replicas. Only writes (new logins) would fail briefly.

[Mike Torres]
That lines up with what I'm thinking. Going to benchmark a Redis session store this weekend and write up the proposal.

One thing I'd recommend — benchmark not just read/write latency but also the Django integration. We used `django-redis` as the session backend and it was mostly plug-and-play. The `SESSION_ENGINE` setting just points to the Redis backend and everything else (session middleware, CSRF, etc.) works as-is. The migration was surprisingly undramatic.

[David Okafor]
Yeah, the Django session framework abstracts the backend really well. The main work is standing up and operating Redis itself — provisioning, monitoring, alerting, backup. The application code changes are minimal.
