**Proposal: Redis caching layer with a 5-minute TTL for API responses.**

Here's the architecture:

```
Client → API Server → Cache Check (Redis) → Hit? → Return cached response
                                           → Miss? → Execute query → Store in Redis (5-min TTL) → Return response
```

**Cache key structure:**
```
cache:{endpoint}:{tenant_id}:{hash_of_query_params}
```

The tenant_id in the key ensures tenant isolation — Tenant A can never receive Tenant B's cached data. The hash is SHA-256 of the sorted query parameters, so `?start=2025-01-01&end=2025-06-01` and `?end=2025-06-01&start=2025-01-01` produce the same cache key.

[Tomas Eriksson — Zoom — 10:12 AM]
David, just to clarify — are you hashing the full query string or the normalized parameters? I've seen bugs where URL encoding differences cause cache misses for semantically identical requests.

[David Okafor — 10:12 AM]
Good point. I'm normalizing first — lowercasing keys, sorting alphabetically, then hashing the canonical form. So `?Start=2025-01-01` and `?start=2025-01-01` hit the same cache entry. Here's the normalization function:

```python
def normalize_cache_params(params: dict) -> str:
    """Produce a canonical string from query params for cache key hashing."""
    normalized = {k.lower(): str(v) for k, v in sorted(params.items())}
    return json.dumps(normalized, sort_keys=True, separators=(',', ':'))
```

**Cache invalidation:**
- TTL-based expiry is the primary mechanism (simple, reliable)
- Explicit invalidation on write: when a dashboard is updated, we invalidate all cache keys for that dashboard
- Bulk invalidation: `cache:flush:{tenant_id}:*` pattern delete for tenant-level cache reset
- No cache warming — we'll rely on the first request after expiry to populate the cache (miss penalty is acceptable)

For the explicit invalidation, I'm using Redis `SCAN` with pattern matching rather than `KEYS` to avoid blocking Redis on large keyspaces. The invalidation happens asynchronously — the write endpoint returns immediately and fires a background task to clear cache entries.

```python
async def invalidate_dashboard_cache(tenant_id: str, dashboard_id: str):
    """Invalidate all cache entries for a specific dashboard."""
    pattern = f"cache:dashboards/{dashboard_id}/*:{tenant_id}:*"
    cursor = 0
    while True:
        cursor, keys = await cache_redis.scan(cursor, match=pattern, count=100)
        if keys:
            await cache_redis.delete(*keys)
        if cursor == 0:
            break
```

[James Holden — 10:36 AM]
What about cache stampede? If a popular dashboard's cache expires and 50 users request it simultaneously, we'd get 50 cache misses all hitting the database at once.

[David Okafor — 10:36 AM]
Good catch. I'm implementing a cache-aside pattern with a distributed lock. When a cache miss occurs, the first request acquires a lock (Redis SETNX with a 5-second TTL), executes the query, and populates the cache. Subsequent requests during the lock period wait briefly (100ms) and then check the cache again. If the cache is populated, they get the cached response. If the lock times out without populating the cache, they fall through to the database.

```python
def get_cached_or_compute(cache_key, compute_fn, ttl):
    # Try cache first
    cached = redis.get(cache_key)
    if cached:
        return deserialize(cached)

    # Try to acquire computation lock
    lock_key = f"lock:{cache_key}"
    acquired = redis.set(lock_key, "1", nx=True, ex=5)

    if acquired:
        # We got the lock — compute and cache
        try:
            result = compute_fn()
            redis.set(cache_key, serialize(result), ex=ttl)
            return result
        finally:
            redis.delete(lock_key)
    else:
        # Someone else is computing — wait and check cache
        time.sleep(0.1)
        cached = redis.get(cache_key)
        if cached:
            return deserialize(cached)
        # Lock holder might have failed — compute directly
        return compute_fn()
```

The lock TTL of 5 seconds is a safety net. If the lock holder crashes or takes too long, the lock auto-expires and the next requester can take over. We never get stuck in a permanent lock state.

[David Okafor — 10:40 AM]

ITEM 2: DASHBOARD WIDGET PERFORMANCE UPDATE (10:48–11:05 AM)

[David Okafor — 10:48 AM]
Quick update on the widget performance ticket (FALCON-09). Two separate improvements:

First, the composite index fix I mentioned in standup. I added a composite index on `(tenant_id, event_type, created_at)` to the events table. The aggregation query for widgets was doing a full scan on 48M rows because the existing index only covered `(tenant_id, created_at)` and the query filters on `event_type` too. After adding the index:

Before:
```
EXPLAIN ANALYZE: Seq Scan on events → 2,340ms, rows=48,000,000
```

After:
```
EXPLAIN ANALYZE: Index Scan using idx_events_tenant_type_created → 89ms, rows=12,400
```

That's a 26x improvement on the database side. Combined with the caching layer, dashboard widgets should go from 2.8 seconds to sub-100ms for cached responses and ~150ms for cache misses.

[Tomas Eriksson — Zoom — 10:51 AM]
David, did adding that index have any impact on write performance? We're doing a lot of INSERTs into the events table.

[David Okafor — 10:52 AM]
Minimal. I checked the INSERT latency before and after: P95 went from 1.2ms to 1.4ms. The index is on three columns but they're all narrow types — UUID, varchar(50), and timestamp. The B-tree maintenance cost is negligible compared to the query improvement. I ran the migration during our low-traffic window on Wednesday night and the index creation took about 3 minutes on the 48M row table using `CREATE INDEX CONCURRENTLY` so there was no table lock.

[Sarah Kim — 10:53 AM]

[David Okafor — 10:53 AM]
Second, I refactored the widget query builder to push aggregation down to the database instead of fetching raw rows and aggregating in Python. The old code was doing:

```python
# Old: fetch all events, aggregate in Python
events = Event.objects.filter(tenant=tenant, type=event_type, created__range=(start, end))
result = {}
for event in events:
    key = event.created.date()
    result.setdefault(key, []).append(event.value)
# ... compute averages, sums, etc.
```

New code:
```python
# New: aggregate in PostgreSQL
result = (
    Event.objects
    .filter(tenant=tenant, type=event_type, created__range=(start, end))
    .values(day=TruncDate('created'))
    .annotate(avg_value=Avg('value'), sum_value=Sum('value'), count=Count('id'))
    .order_by('day')
)
```

This reduces data transfer from database to application by ~99% for large date ranges. Instead of fetching 50K rows and aggregating in Python, we get back 30 pre-aggregated rows.

That's a classic optimization. I'm surprised it wasn't done that way originally.

[David Okafor — 10:56 AM]
Legacy code. The original implementation was written when the events table had 10K rows and the Python aggregation was fast enough. Nobody revisited it as the table grew. This is one of those gradual degradation issues — it worked fine for a year and then suddenly it didn't.
