ITEM 3: INCIDENT REVIEW — STAGING REDIS CONNECTIVITY (10:52–11:10 AM)

[Carlos Rivera — 10:52 AM]
Minor incident to review. Yesterday at 4:17 PM, the staging session Redis became unreachable for 47 seconds. Here's what happened:

Root cause: ElastiCache performed an automatic failover test in the staging environment. This is a feature I enabled for staging to verify our failover behavior. The configuration is `--automatic-failover-enabled` with `test-failover` triggered via a scheduled CloudWatch event every Tuesday at 4 PM. During the failover, the primary node was replaced and the DNS endpoint updated. Our application's Redis connection pool had stale connections pointing to the old primary.

Timeline:
- 16:17:00 — ElastiCache initiates failover
- 16:17:02 — Primary node begins shutdown
- 16:17:15 — New primary promoted from replica
- 16:17:22 — DNS endpoint updated to new primary IP
- 16:17:47 — Application establishes new connections, service restored

Impact: During the 47-second window, session lookups in staging returned errors. The session middleware correctly fell back to the PostgreSQL backend for those requests — no user-visible impact. We saw 234 fallback events in the staging logs during that window.

The production Redis cluster has the same failover configuration but we haven't triggered it yet. Based on this staging test, I need to add connection health checking to our Redis pool — on connection error, discard the connection and establish a new one. The Python `redis-py` library supports this with the `health_check_interval` parameter, but we're using a custom connection pool wrapper that doesn't pass it through. I need to update the wrapper.

[Mike Torres — 10:56 AM]
The session middleware handled it correctly — it fell back to PostgreSQL. But once we remove the PostgreSQL fallback (Phase 2), a Redis outage would mean session lookups fail and users get 401s. How do we handle that?

[Carlos Rivera — 10:57 AM]
Two things. First, the connection health check I mentioned — after failover, new connections automatically point to the new primary via DNS. So the outage window is just the time it takes to establish new connections, not the full failover time. With health checking, that's sub-second. The `health_check_interval` parameter sends a PING before each command if the connection has been idle for more than N seconds. If the PING fails, it drops the connection and creates a new one.

Second, the ElastiCache Multi-AZ failover in production typically completes in 10-15 seconds. During that window, reads can go to the replica (I'll configure read replicas for session lookups). Writes (new session creation, token rotation) would fail briefly, but that only affects new logins and active rotations — existing sessions would get a stale token (old one still valid) and rotate on the next request.

[Sarah Kim — 10:58 AM]
Does the export worker also use the session Redis, or is it only the API service?

[Carlos Rivera — 10:58 AM]
Only the API service and the WebSocket service use the session Redis. The export worker authenticates via a service account token that's validated against the database directly — it doesn't go through the session middleware. So a session Redis outage wouldn't affect exports at all.

[Mike Torres — 10:59 AM]
That's acceptable. The worst case is a 15-second window where new logins fail and active users get a brief delay on token rotation. No data loss, no session invalidation.

[David Okafor — 11:00 AM]
For the cache Redis (the separate cluster for API caching), failure is even more benign — cache miss, database query, user sees normal latency instead of cached latency. No auth implications. And the cache Redis is a single-node setup without Multi-AZ, so its failure mode is just "node dies, we provision a new one, cache is cold." The application handles that gracefully because every cache read has a database fallback.

[Carlos Rivera — 11:01 AM]
Exactly. I want to stress again that these are separate clusters with separate failure domains. A session Redis failure does NOT affect cache Redis and vice versa. They're in different ElastiCache replication groups, different subnet groups, and the application uses different connection pool instances for each. There's no shared state between them.

[James Holden — 11:02 AM]
Should we document this failure mode in the incident runbook? It seems like a scenario the on-call person should know about.

[Carlos Rivera — 11:02 AM]
Good idea. I'll add a section on Redis failover handling — expected duration, what the user impact looks like, and what the on-call response should be (which is mostly "wait for automatic recovery and verify connections re-establish").

[Priya Chandrasekaran — 11:02 AM]
Good. Carlos, file a ticket for the connection health checking fix and get it in before we remove the PostgreSQL fallback.

[Carlos Rivera — 11:03 AM]
Already filed. SPRINT25-INFRA-004. I'll have a PR up by Friday. The fix itself is small — it's a one-line change to pass `health_check_interval=10` to the connection pool constructor, plus updating the wrapper class to accept the parameter.
