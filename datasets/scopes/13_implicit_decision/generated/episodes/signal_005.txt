## Set up Redis session store with token rotation

### Summary

Implements a Redis-backed session store to replace the current PostgreSQL `django_session` backend. Sessions are stored as Redis keys with built-in TTL for automatic expiry. Includes token rotation on each request for security hardening.

### Background

Our PostgreSQL session store is causing user-facing issues (SUP-4398–4402, SUP-4421, SUP-4423):
- Connection pool contention during peak load causes session lookup timeouts
- Session cleanup cron job had a timezone bug (fixed in #1858) but the cron-based approach is fragile
- Session table (15K+ rows) competing with application queries for pool connections

See ADR-017 for full analysis.

### What This PR Does

**New session middleware** (`rivulet/auth/session_middleware.py`):
- Replaces `django.contrib.sessions.middleware.SessionMiddleware`
- On each request: reads session token from cookie, performs Redis GET to retrieve session data
- If session is valid: deserializes session data, attaches `user_id` and `tenant_id` to request, rotates token
- If session is invalid/expired: returns 401 (API) or redirects to login (browser)
- Token rotation: generates new 256-bit token, SET with TTL, DEL old token — atomic via Lua script
- Exempted paths configurable via `SESSION_EXEMPT_PATHS` (defaults to `/api/health`, `/api/auth/login`, `/api/auth/signup`)
- Middleware ordering: runs after `CorsMiddleware`, before `TenantContextMiddleware`

**Session manager** (`rivulet/auth/session_manager.py`):
- `create_session(user_id, tenant_id, metadata)` → creates Redis entry, returns session token
- `get_session(token)` → retrieves session data, returns None if expired/missing
- `rotate_token(old_token)` → atomic rotate: generate new token, copy data, set TTL, delete old
- `revoke_session(token)` → DEL the key. Immediate invalidation.
- `revoke_all_user_sessions(user_id)` → scans for all sessions for a user, DELs them all
- `get_active_session_count(user_id)` → returns number of active sessions for a user
- All public methods log structured events via `structlog` with `action`, `user_id`, `session_token_prefix` (first 8 chars only)

**Redis connection management** (`rivulet/auth/redis_pool.py`):
- Dedicated connection pool for session operations (separate from rate limiter pool)
- Pool size: 20 connections (configurable via `SESSION_REDIS_POOL_SIZE`)
- Connection timeout: 100ms (fast-fail if Redis is unresponsive)
- Retry: 1 retry with 50ms backoff
- Health check interval: 30 seconds (`health_check_interval=30` on the pool)
- Socket keepalive enabled to prevent idle connection drops behind NAT/firewalls

**Token format**:
- 256-bit random bytes, base64url-encoded → 43 characters
- Prefix: `rsess_` for easy identification in logs
- Example: `rsess_a7Kx9mP2qL4nR8vT1wB5yC3dF6hJ0sU2eG4iM7o`
- Redis key format: `session:{token}` with configurable key prefix
- Token generation uses `secrets.token_urlsafe(32)` — cryptographically secure

**Session data structure** (stored as JSON in Redis):
```json
{
  "user_id": "usr_a1b2c3d4",
  "tenant_id": "tnt_x9y8z7w6",
  "created_at": "2025-06-11T10:30:00Z",
  "last_rotated_at": "2025-06-11T10:45:00Z",
  "ip_address": "192.168.1.100",
  "user_agent": "Mozilla/5.0...",
  "metadata": {}
}
```

**Session lifecycle**:
- Default TTL: 24 hours (configurable via `SESSION_TTL_SECONDS`)
- Token rotation resets TTL → active users never expire
- Idle sessions expire naturally via Redis TTL
- Max sessions per user: 10 (configurable via `SESSION_MAX_PER_USER`) — creating an 11th revokes the oldest

**Security features**:
- Token rotation on every request (stolen tokens are single-use)
- Duplicate token detection: if a rotated-out token is used, all sessions for that user are revoked (indicates possible token theft)
- Session binding: IP address and user agent are stored and optionally validated (strict mode for enterprise)
- Rate limiting on login endpoint: 5 attempts per minute per IP (uses existing rate limiter)
- Tokens never appear in logs — only the first 8 characters are logged (e.g., `rsess_a7K*****`)

### Lua Scripts

Token rotation is implemented as an atomic Lua script to prevent race conditions:

```lua
-- rotate_token.lua
-- Atomically rotate a session token
-- KEYS[1] = old session key
-- KEYS[2] = new session key
-- ARGV[1] = TTL in seconds

local data = redis.call('GET', KEYS[1])
if not data then
    return nil  -- old token already expired/revoked
end

-- Set new key with data and TTL
redis.call('SET', KEYS[2], data, 'EX', ARGV[1])
-- Delete old key
redis.call('DEL', KEYS[1])
-- Track rotation for duplicate detection
redis.call('SET', KEYS[1] .. ':rotated', KEYS[2], 'EX', 300)

return data
```

The `:rotated` suffix key is used for duplicate detection. If a request comes in with a token that has been rotated out, we check if the `:rotated` key exists. If it does, someone is using an old token — possible session theft. We revoke all sessions for that user as a precaution.

**Duplicate detection script** (`duplicate_check.lua`):
```lua
-- duplicate_check.lua
-- Check if a token was previously rotated out
-- KEYS[1] = old session key + ':rotated'
-- Returns: the new token key if rotated, nil otherwise

local rotated_to = redis.call('GET', KEYS[1])
if rotated_to then
    -- Extract user_id from the new session to revoke all
    local new_data = redis.call('GET', rotated_to)
    return new_data
end
return nil
```

### Rollback Plan

If Redis sessions cause issues in staging or production:
1. Set feature flag `redis_sessions` to OFF — all new logins go back to Django sessions
2. Existing Redis sessions will still work until they expire (24h max)
3. No data migration needed — PostgreSQL session table is untouched during Phase 1

The `create_session` method — I like that you're using `SET ... NX` to prevent token collisions. The probability of a collision with 256-bit tokens is astronomically low, but defense in depth is always good.
