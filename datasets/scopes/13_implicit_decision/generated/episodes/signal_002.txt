[Sarah Kim — 2:11 PM]
Right. So given those problems, I want to propose we move to JWT-based authentication. Here's my reasoning.

[Slide: "Proposal: JWT (JSON Web Tokens)"]

JWT fundamentally changes the model. Instead of storing sessions server-side and looking them up on every request, we encode the session data INTO the token itself. The server signs the token with a secret key, and on each request, it just verifies the signature — no database lookup at all.

For anyone not familiar with the format, a JWT is three base64-encoded segments separated by dots: `header.payload.signature`. The header specifies the algorithm, the payload contains claims (key-value pairs with the session data), and the signature is a cryptographic hash that proves the payload hasn't been tampered with.

Benefits:
- **Stateless**: Zero server-side storage for sessions. The token IS the session.
- **Horizontal scaling**: Any server instance can validate any token because it just needs the signing key. No shared session store needed. We could go from 3 API servers to 30 and sessions would just work.
- **Performance**: Token verification is a cryptographic operation — sub-millisecond. No database query, no connection pool usage, no I/O at all.
- **No cleanup job**: Tokens have built-in expiry (`exp` claim). They're self-invalidating. The timezone bug goes away entirely because the client is responsible for expiry enforcement.
- **CDN-friendly**: Since tokens are self-contained, we could push auth validation to the edge if we ever go that route.

[Lin Zhang — 2:14 PM]
Quick question on the scaling point — when you say any server instance can validate, does that mean we could remove the ALB session affinity config?

[Sarah Kim — 2:14 PM]
Exactly. Right now we have sticky sessions enabled on the ALB because the session cookie has to route to the same backend that has the PostgreSQL connection. With JWT, there's no affinity needed — any backend can validate any token because they all share the same public key. That's a meaningful simplification for the infra team too.

OK, moving on to the implementation details. I'm thinking:
- Short-lived access tokens: 15-minute expiry, signed with RS256
- Long-lived refresh tokens: 7-day expiry, stored in HttpOnly cookies
- Token payload: `sub` (user ID), `tid` (tenant ID), `roles` (array), `iat`, `exp`
- Refresh flow: client POSTs to `/api/auth/refresh` with the refresh token cookie, gets a new access token

Here's a concrete example of what the token payload looks like:

```json
{
  "sub": "usr_7f3a2b1c",
  "tid": "tnt_haverfield",
  "roles": ["analyst", "dashboard_editor"],
  "iat": 1717444800,
  "exp": 1717445700,
  "jti": "tok_9d8e7f6a5b4c"
}
```

That encodes to about 400 bytes total in the JWT. I also sketched out the middleware. It's actually simpler than what we have now:

```python
class JWTAuthMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response
        self.public_key = load_rsa_public_key(settings.JWT_PUBLIC_KEY_PATH)

    def __call__(self, request):
        token = self._extract_token(request)
        if token:
            try:
                payload = jwt.decode(
                    token,
                    self.public_key,
                    algorithms=["RS256"],
                    options={"require": ["exp", "sub", "tid"]}
                )
                request.user_id = payload["sub"]
                request.tenant_id = payload["tid"]
                request.user_roles = payload.get("roles", [])
            except jwt.ExpiredSignatureError:
                return JsonResponse({"error": "token_expired"}, status=401)
            except jwt.InvalidTokenError:
                return JsonResponse({"error": "invalid_token"}, status=401)
        return self.get_response(request)
```

No database call. No session table. No connection pool usage for auth. It's clean.

[Lin Zhang — 2:22 PM]
What about the frontend? Right now we're sending the session cookie and Django handles everything. With JWT we'd need to change the login flow, add token storage, handle refresh...

[Sarah Kim — 2:22 PM]
The access token goes in the Authorization header as a Bearer token. The refresh token stays in an HttpOnly cookie — same mechanism as the current session cookie, so the browser handles it automatically. The main frontend change is:
1. Login endpoint returns the access token in the response body (and sets the refresh cookie)
2. Frontend stores the access token in memory (NOT localStorage — that's an XSS vector)
3. Frontend adds `Authorization: Bearer <token>` to every API request
4. When a 401 comes back, frontend hits the refresh endpoint before retrying

I can write the frontend auth service. It's maybe 150 lines of TypeScript. The core of it is an Axios interceptor:

```typescript
// authService.ts
let accessToken: string | null = null;

api.interceptors.request.use((config) => {
  if (accessToken) {
    config.headers.Authorization = `Bearer ${accessToken}`;
  }
  return config;
});

api.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401 && !error.config._retry) {
      error.config._retry = true;
      const { data } = await api.post('/api/auth/refresh');
      accessToken = data.access_token;
      error.config.headers.Authorization = `Bearer ${accessToken}`;
      return api(error.config);
    }
    return Promise.reject(error);
  }
);
```

[Priya Chandrasekaran — 2:27 PM]
I like the direction. It solves all three of our immediate problems — the cleanup bug, the connection pool issue, and the scaling concern. What are the risks?

[Sarah Kim — 2:27 PM]
Main risk is key management. If the RSA private key is compromised, all tokens are compromised. But we'd use AWS KMS for key storage and rotation, which is what we already do for our encryption-at-rest keys. So it's a solved problem operationally.

The key rotation flow would be:
1. Generate new key pair in KMS
2. Deploy new private key for signing, add new public key to the verification set
3. New tokens signed with new key; old tokens still verified by old key
4. After the longest token lifetime (7 days for refresh tokens), remove the old public key

We already have this pattern implemented for our API signing keys in the partner integration layer.

Second risk is token size growth. If we start putting too much in the payload, the tokens get large. But that's a code review issue, not an architectural one. We can set a lint rule or a middleware check that rejects tokens over a certain size.

[Sarah Kim — 2:36 PM]
You could use a token blacklist. On revocation, add the token's `jti` (unique ID) to a blacklist. Check the blacklist on each request.

[Mike Torres — 2:36 PM]

ADR-017 DRAFT (committed to architecture repo, 5:45 PM)

# ADR-017: Session Management Architecture

## Status
PROPOSED — under discussion

## Context
The current PostgreSQL-backed session store is causing user-facing issues:
- Timezone bug in cleanup job deleting valid sessions
- Connection pool contention causing session lookup timeouts during peak load
- Session table growing linearly (15K+ rows) with no horizontal scaling path

8 support tickets (SUP-4398–4402, SUP-4421, SUP-4423) in 3 weeks. Two enterprise customers (TritonMetrics, Haverfield Analytics) have flagged as P1.

## Options Under Consideration

### Option A: JWT-Based Authentication
- Proposed by: Sarah Kim
- See detailed analysis in meeting notes (2025-06-03)
