**Item: Stop writing test descriptions in comments — use descriptive test names instead**
Written by: Yuto Hayashi

[Yuto Hayashi — 3:38 PM]
I was reviewing Marcus's reconciliation tests and found a lot of this pattern:

```python
def test_retry_case_1(self):
    """Test that a failed reconciliation with a transient error is retried
    up to 3 times with exponential backoff."""
    ...

def test_retry_case_2(self):
    """Test that a failed reconciliation with a permanent error is
    not retried and goes directly to the dead-letter queue."""
    ...
```

When the test fails in CI, you see `FAILED test_retry_case_1` — which tells you nothing. If the test was named `test_transient_error_retries_three_times_with_exponential_backoff`, the failure message is self-documenting.

[Marcus Webb — 3:39 PM]
Fair point. I was being lazy. The comment has the information but the test name doesn't.

[Angela Fong — 3:40 PM]
I had the same issue last sprint when a test failed in the partner API suite while I was on-call. The test was called `test_edge_case_3`. I had to open the file and read the docstring to understand what it was even testing. At 2am that's not fun.

[Derek Lane — 3:40 PM]
This is a code review issue. We should be catching this in reviews. But it's also a convention issue — if the convention is clear, people will follow it.

[Priti Sharma — 3:41 PM]
There's also the issue of test output readability. When you run pytest with verbose mode, you get a list of test names. If those names are descriptive, the test output reads like a specification of the system's behavior. `test_case_1, test_case_2, test_case_3` reads like nothing.

[Simone Osei — 3:41 PM]
Can we add a linting rule for minimum test name length? Something like "test function names must have at least 5 words separated by underscores."

[Angela Fong — 3:42 PM]
Five words is arbitrary. I'd rather focus on the principle: test names should describe the behavior being tested, not just "case 1, case 2."

[Yuto Hayashi — 3:42 PM]
The principle is what matters. But having a concrete example helps people follow it. Something like: instead of `test_validate_input`, write `test_validate_input_rejects_negative_amounts`. Instead of `test_retry_logic`, write `test_retry_exhausted_sends_to_dead_letter_queue`.

[Marcus Webb — 3:43 PM]
I'll go back and rename the reconciliation tests this sprint. Should only take an hour or so and it'll make the test suite much more readable.

[Derek Lane — 3:43 PM]
You don't need to do that as a separate task — just rename them when you're next touching those files. Boy Scout rule.

[Kai Nakamura — 3:43 PM]
**Decision: We decided to adopt a convention of descriptive test names that describe the tested behavior.** Format: `test_<condition>_<expected_behavior>`. Reviewers should flag generic test names (e.g., `test_case_1`) in code reviews. No automated linting for now — we'll revisit if compliance is low.

[Kai Nakamura — 3:48 PM]

[Priti Sharma — 3:49 PM]

[Simone Osei — 3:50 PM]

[Kai Nakamura — 3:50 PM]

**Item: Continue design reviews for 5+ point tickets**
Written by: Derek Lane, Angela Fong, Simone Osei

[Derek Lane — 3:52 PM]
The design review policy from last sprint's retro was the single biggest improvement this sprint. The payment reconciliation design review caught 3 edge cases:
1. What happens when a retry succeeds but the original request also eventually succeeds? (duplicate processing)
2. What happens when the dead-letter queue is full? (backpressure)
3. What happens when the state machine receives an event for a state it's already transitioned out of? (stale event)

Each of these would have been a production bug. Instead, we handled them in the design phase and the implementation was clean.

[Marcus Webb — 3:53 PM]
From the implementer side, the design review also gave me confidence that I wasn't missing something. When Derek and I white-boarded the state transitions, I could see all the paths clearly. Without that, I would have discovered the stale event issue during testing, probably after already building the wrong thing.

[Priti Sharma — 3:54 PM]
How long did the three edge case fixes take to implement once they were identified in the design review?

[Marcus Webb — 3:54 PM]
The duplicate processing guard was about 2 hours — it was an idempotency check using the reconciliation ID. The backpressure handling was maybe 3 hours — we added a circuit breaker on the dead-letter queue producer. The stale event handling was the simplest — just an event sequence number check, maybe an hour. So about 6 hours total for all three, versus probably 2 days of debugging if we'd found them in production.

[Angela Fong — 3:54 PM]
The partner onboarding design review simplified the key provisioning flow. We went from a 5-step async flow to a 3-step sync flow that's easier to understand, test, and debug. The 30-minute review saved probably a week of implementation time.

[Simone Osei — 3:55 PM]
The original 5-step flow had this unnecessary async handoff between key generation and key storage — we were publishing to a queue and then consuming from it in the same service. In the design review, Derek asked "why is this async?" and the honest answer was "because I copied the pattern from the notification service." The notification service has a good reason to be async (external email provider). The key provisioning doesn't.

[Derek Lane — 3:55 PM]
That's a great example of how design reviews prevent accidental complexity. It's not that the 5-step flow was wrong — it would have worked. But the 3-step flow is simpler, has fewer failure modes, and is easier to reason about.

[Simone Osei — 3:55 PM]
From the reviewer side, it's also a great knowledge-sharing mechanism. I now understand the payment reconciliation system well enough to debug it on-call, even though I didn't build it. That was a free bonus of the design review.

[Yuto Hayashi — 3:56 PM]
Same here. I attended the partner onboarding review as an observer. Even though I didn't contribute much, I now know how the key provisioning works, what the database schema looks like, and where the logs go. That's on-call readiness without explicit training.
