ITEM 1: API VERSIONING STRATEGY (3:00–3:42 PM)

[Sarah Kim — 3:00 PM]
Thanks everyone. So this has been on the backlog for a while and the multi-workspace work is going to force our hand. When we add workspace_id to API endpoints, we're making a breaking change to the API contract. We need a versioning strategy before we can ship multi-workspace to customers.

Let me walk through the options and then propose a direction. I put a lot of thought into this over the weekend so bear with me — there's a lot of ground to cover.

**Option 1: URL path versioning**
```
GET /api/v1/dashboards
GET /api/v2/dashboards?workspace_id=ws_abc123
```
Pros: Explicit, easy to route, easy to document, clients know exactly which version they're hitting. Every request wears its version on its sleeve. You can see it in access logs, in curl commands, in browser dev tools. No ambiguity.
Cons: URL proliferation, harder to share links (which version?), some REST purists consider it an anti-pattern because the version isn't part of the resource's identity.

**Option 2: Header-based versioning**
```
GET /api/dashboards
Accept: application/vnd.rivulet.v2+json
```
Pros: Clean URLs, RESTful, version is metadata not resource identity. GitHub's API actually supports this pattern as an alternative to their path versioning.
Cons: Harder to test (need to set headers), harder to document, CDN caching complications, some clients don't easily support custom Accept headers. I tested this with Postman and it's fine, but a lot of enterprise customers are using simple curl scripts or low-code tools like Zapier that don't have easy header customization.

**Option 3: Query parameter versioning**
```
GET /api/dashboards?api_version=2
```
Pros: Easy to test, explicit, works with any client.
Cons: Pollutes the query string, easy to forget, mixed with business parameters. Also creates weird interactions with our existing pagination and filter params — you'd see `?api_version=2&page=3&sort=created_at` and it's messy.

[Tom Nguyen — 3:08 PM]
Quick question — did you look at date-based versioning? I've seen some APIs use dates like `2025-06-15` instead of v1/v2.

[Sarah Kim — 3:09 PM]
I did look at that. Stripe actually uses date-based versioning for their changelog — like `Stripe-Version: 2023-10-16`. The problem is it creates an explosion of versions. Every breaking change is a new version date, and you end up maintaining dozens of effective versions. It works for Stripe because they have a dedicated API compatibility team. We don't have that luxury. I think major version numbers are more manageable for a team our size.

[Tom Nguyen — 3:10 PM]
Fair enough. Thanks for looking into it.

[Sarah Kim — 3:10 PM]
My recommendation: **Option 1, URL path versioning.** Here's why:

1. Our customers are enterprise B2B. They build integrations against our API. Explicit versioning in the URL makes it impossible to accidentally hit the wrong version. When they're debugging, the version is right there in the URL.

2. We're building a customer-facing API — not an internal microservice. Discoverability matters more than REST purity.

3. Stripe, GitHub, and Twilio all use URL path versioning. Our customers are used to it.

4. It simplifies our routing layer. We can route `/api/v1/*` and `/api/v2/*` to different handler modules without any middleware magic.

[Elena Marchetti — 3:13 PM]
From the docs side, I strongly prefer path versioning. It makes the documentation structure much cleaner — we can have separate doc sections for v1 and v2, and customers can see exactly which version they're reading about in the URL. With header-based versioning, documenting the differences becomes really confusing.

[Sarah Kim — 3:14 PM]
Great point, Elena. That's another practical advantage I should have included.

**Implementation plan:**

Phase 1: Add v1 prefix to all existing endpoints. Deploy as a transparent proxy — `/api/v1/dashboards` routes to the same handler as `/api/dashboards`. The old unprefixed URLs continue to work but return a `Deprecation` header.

I'm thinking the deprecation header would look like this:
```
Deprecation: true
Sunset: 2026-06-15
Link: <https://docs.rivulet.io/api/migration/v1>; rel="deprecation"
```
That follows RFC 8594 for the Sunset header and RFC 8288 for the Link header. Gives clients machine-readable migration info.

Phase 2: Build v2 endpoints for multi-workspace. These require workspace_id and have updated response schemas. v1 endpoints continue to work for tenants that haven't adopted workspaces (they hit the "default" workspace implicitly).

For Phase 2, the v2 response schema changes look like this for the dashboards endpoint:
```json
// v1 response
{
  "id": "dash_abc123",
  "name": "Revenue Overview",
  "widgets": [...]
}

// v2 response
{
  "id": "dash_abc123",
  "name": "Revenue Overview",
  "workspace_id": "ws_xyz789",
  "workspace_name": "Finance Team",
  "widgets": [...]
}
```

Phase 3 (6 months out): Deprecation notice for v1. Give customers 6 months to migrate. Remove v1 after 12 months.

**Versioning policy (proposed):**
- New major version only for breaking changes (removed fields, changed types, new required parameters)
- Additive changes (new optional fields, new endpoints) are added to the current version
- We maintain at most 2 major versions simultaneously
- Each version has a minimum 12-month support window after deprecation notice

[Mike Torres — 3:15 PM]
I'm on board with URL path versioning. Quick question on the routing — are you thinking separate Django URL configs per version, or a middleware that rewrites the path?

[Sarah Kim — 3:16 PM]
Separate URL configs. `urls_v1.py` and `urls_v2.py`, each included under their prefix in the main `urls.py`. This gives us clean separation — v1 and v2 handlers can live in different modules without any routing magic.

```python

    path('api/v1/', include('rivulet.api.v1.urls')),
    path('api/v2/', include('rivulet.api.v2.urls')),
    # Legacy unprefixed routes (deprecated)
    path('api/', include('rivulet.api.v1.urls')),
]
```

For shared logic, the view code lives in a common module and the version-specific URL configs point to thin wrapper views that handle schema differences.

ITEM 2: MULTI-WORKSPACE DATABASE MIGRATION PLAN (3:42–4:02 PM)

[David Okafor — 3:42 PM]
I've drafted the database migration plan for multi-workspace. Let me walk through the approach.

The goal: add `workspace_id` to all tenant-scoped tables, with data isolation guarantees.

**Tables affected** (in migration order):
1. `workspaces` — new table: (id, tenant_id, name, created_at, updated_at, is_default)
2. `dashboards` — add workspace_id FK
3. `data_sources` — add workspace_id FK
4. `events` — add workspace_id FK (the big one: 48M rows)
5. `export_jobs` — add workspace_id FK
6. `permissions` — restructure to include workspace_id scope
7. `notifications` — add workspace_id FK

[Tom Nguyen — 3:45 PM]
Forty-eight million rows — how big is the events table on disk right now?

[David Okafor — 3:45 PM]
About 14GB. The UUID column add is metadata-only in PostgreSQL so it won't rewrite the table. The backfill writes are the expensive part, but they're batched. I'll get to the numbers in a minute.

[Elena Marchetti — 3:46 PM]
Quick question — the `permissions` table restructure, is that also an additive change or is it more involved?

[David Okafor — 3:46 PM]
More involved. Currently permissions are tenant-scoped: a user has a role within a tenant. With workspaces, we need workspace-scoped permissions: a user has a role within a workspace within a tenant. I'm adding a `workspace_id` column to the permissions table, and the application code will check workspace-level permissions. Existing tenant-level permissions become "all workspaces" permissions via a default workspace assignment. I'll cover the details in a separate design doc — today I want to focus on the events table since that's the riskiest part.

**Migration strategy for the events table:**

Step 1: Create the `workspaces` table and auto-create a "Default" workspace for each existing tenant.

The migration for this step looks like:
```sql
CREATE TABLE workspaces (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id),
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    is_default BOOLEAN NOT NULL DEFAULT false,
    UNIQUE(tenant_id, name)
);

CREATE INDEX idx_workspaces_tenant ON workspaces(tenant_id);

-- Auto-create default workspace for each existing tenant
INSERT INTO workspaces (tenant_id, name, is_default)
SELECT id, 'Default', true FROM tenants;
```

Step 2: Add `workspace_id` as a NULLABLE column to `events`. This is a metadata-only change in PostgreSQL — no table rewrite, instant.
```sql
ALTER TABLE events ADD COLUMN workspace_id UUID;
```

Step 3: Backfill `workspace_id` with the default workspace ID for each tenant. Batched UPDATE, 10K rows per batch, throttled to avoid lock contention.
```sql
UPDATE events SET workspace_id = (
    SELECT id FROM workspaces WHERE tenant_id = events.tenant_id AND is_default = true
)
WHERE workspace_id IS NULL
LIMIT 10000;
```
Estimated time: 48M rows ÷ 10K per batch × 50ms per batch = ~4 minutes. (PostgreSQL batch updates are fast when the source is indexed.)

[Mike Torres — 3:50 PM]
Are you running the backfill in a Django management command or as raw SQL?

[David Okafor — 3:50 PM]
Management command. That way we get logging, progress output, and we can pause/resume if needed. I'll add a `--batch-size` flag and a `--sleep-between-batches` flag for throttling. Something like:

```python
# management/commands/backfill_workspace_id.py
class Command(BaseCommand):
    def add_arguments(self, parser):
        parser.add_argument('--batch-size', type=int, default=10000)
        parser.add_argument('--sleep-ms', type=int, default=0)
        parser.add_argument('--dry-run', action='store_true')

    def handle(self, **options):
        batch_size = options['batch_size']
        total_updated = 0
        while True:
            with connection.cursor() as cursor:
                cursor.execute("""
                    UPDATE events SET workspace_id = (
                        SELECT id FROM workspaces
                        WHERE tenant_id = events.tenant_id AND is_default = true
                    )
                    WHERE id IN (
                        SELECT id FROM events
                        WHERE workspace_id IS NULL
                        LIMIT %s
                    )
                """, [batch_size])
                updated = cursor.rowcount
            total_updated += updated
            self.stdout.write(f"Updated {total_updated} rows...")
            if updated < batch_size:
                break
            if options['sleep_ms']:
                time.sleep(options['sleep_ms'] / 1000.0)
```

Step 4: Add NOT NULL constraint with NOT VALID option (doesn't scan the table):
```sql
ALTER TABLE events ADD CONSTRAINT events_workspace_id_not_null CHECK (workspace_id IS NOT NULL) NOT VALID;
```

Step 5: Validate the constraint in the background:
```sql
ALTER TABLE events VALIDATE CONSTRAINT events_workspace_id_not_null;
```
This does a full table scan but with a weak lock (SHARE UPDATE EXCLUSIVE) that doesn't block normal operations.

Step 6: Create the composite index concurrently:
```sql
CREATE INDEX CONCURRENTLY idx_events_workspace_type_created ON events (workspace_id, event_type, created_at);
```

[Tom Nguyen — 3:54 PM]
How long does the concurrent index creation take on 48M rows?

[David Okafor — 3:54 PM]
Tested on a staging copy — about 8 minutes for the composite index. It runs fully in the background, doesn't block reads or writes. The only thing it blocks is other DDL on the events table, which shouldn't be a problem since we're not running other migrations simultaneously.

**Zero-downtime guarantee:** Every step uses non-blocking DDL. The application continues serving requests throughout. The only behavioral change is that new events start including workspace_id after the code deploy, and the backfill handles existing data.

[Mike Torres — 3:52 PM]
What happens during the backfill if a new event is inserted without a workspace_id? The NOT NULL constraint isn't enforced yet.

[David Okafor — 3:53 PM]
Good catch. The application code needs to be updated to include workspace_id BEFORE we add the NOT NULL constraint. So the deploy order is:
1. Deploy migration (add nullable column)
2. Deploy application code (include workspace_id in all new events)
3. Run backfill for historical data
4. Add NOT NULL constraint
5. Validate constraint

Steps 2 and 3 can happen in any order since the constraint isn't enforced yet. But step 2 must happen before step 4.

[James Holden — 3:54 PM]

[David Okafor — 3:56 PM]
Right. When the multi-workspace code deploys, we flush the cache. It's a one-time event and the cache repopulates within 5 minutes (the TTL). I'll add a cache flush to the migration script.

- When a matching event occurs, we enqueue a delivery job
- Worker picks up the job, makes the POST request, records the result
- Retry with exponential backoff on failure (1min, 5min, 30min, 2hr)
- After 5 failed attempts, mark the webhook as unhealthy and notify the customer

I'll put together a design doc by next week.
