SECTION 1: INCIDENT REPORT — MAY 23 CONNECTION POOL EXHAUSTION (RESOLVED)

INCIDENT ID: INC-2025-003
SEVERITY: P2 (service degradation, not outage)
DURATION: 22 minutes (May 23, 14:12-14:34 ET)
DETECTED BY: Automated PagerDuty alert (connection pool utilization threshold)
ON-CALL ENGINEER: Amir Patel
AFFECTED USERS: ~15 accounts experienced elevated latency (P95 jumped to 2.1s)

TIMELINE OF EVENTS:

14:07 ET — NovaTech Labs user "daniel.wong@novatechlabs.com" initiated a batch export request through the /v1/export/batch endpoint. The request specified a full historical pull of all contact records with associated activity timelines, covering approximately 18 months of ingested data. The query plan generated by our export planner estimated the job would require scanning ~4.2 million rows across three joined tables (contacts, activities, activity_metadata).

14:12 ET — PagerDuty alert fired: "Connection pool utilization at 62% (threshold: 60%)." Amir acknowledged the alert within 90 seconds. At this point, the NovaTech batch export query had acquired 40 database connections and was performing a sequential scan across the activity_metadata table due to a missing composite index on (account_id, created_at). The query was running against the primary database instance, not the read replica.

14:14 ET — Amir began investigation. Initial triage focused on identifying which account and which query were holding connections. The pg_stat_activity view showed 40 connections in "active" state, all originating from the same application-level session token. The query text confirmed it was the NovaTech batch export.

14:18 ET — Amir messaged Marcus and Jess in the #engineering Slack channel: "Seeing a P2 — NovaTech batch export is holding 40/100 connections, other accounts starting to queue. Going to kill the query and implement a short-term mitigation. Will post updates here." Marcus responded: "Go ahead, keep me posted."

14:22 ET — Connection pool utilization hit 71%. Three other accounts — Clearpoint Financial, Meridian Data Services, and UrbanGrid Analytics — reported elevated query latency in their application logs (we could see this from our server-side request tracing). None of these accounts contacted support. The elevated latency manifested as P95 response times of 2.1 seconds, up from the normal baseline of approximately 150ms.

14:26 ET — Amir executed pg_terminate_backend() against the NovaTech query's backend PID. The 40 connections were released immediately. Connection pool utilization dropped to 23% within 15 seconds. Queued queries from other accounts began processing.

14:34 ET — All latency metrics returned to baseline. Amir marked the incident as resolved in PagerDuty.

14:45 ET — Amir called Daniel Wong at NovaTech Labs to explain the situation. Daniel was understanding: "No worries — we should have chunked that export. Is there a recommended way to do large pulls?" Amir explained the batch export was being re-architectured (see below) and that we would re-submit his export once the fix was in place.

ROOT CAUSE:

A large batch export request from NovaTech Labs (12 seats, heavy usage account) triggered a long-running query that held 40 database connections for 18 minutes. Our connection pool limit is 100 connections. When other accounts' queries arrived simultaneously, the pool was 40% consumed by one query, causing queueing for everyone else.

Contributing factors identified during the post-incident review:
- The batch export code path opens one database connection per partition of the export. NovaTech's export was partitioned into 40 monthly chunks (18 months of data, but the partitioner over-segmented due to uneven data distribution). Each partition held its connection for the full duration rather than using a checkout/return pattern.
- The export query ran against the primary database instead of the read replica. This was a known tech debt item — the export code was written during the pre-launch phase when we only had a single database instance. The read replica was added in week 1 but the export code path was never updated.
- The activity_metadata table was missing a composite index on (account_id, created_at), causing the sequential scan. With the index, the same query would complete in approximately 90 seconds instead of 18+ minutes.

RESOLUTION:
- Immediate: killed the long-running query, connections released
- Permanent fix: implemented per-account connection limits (max 20 connections per account) and query timeout of 60 seconds for batch exports
- Additional fix (Amir, completed May 24): added the composite index on activity_metadata(account_id, created_at). Index creation took 8 minutes during low-traffic window (Saturday 06:00 ET). Verified via EXPLAIN ANALYZE that the NovaTech export query now uses an index scan.
- Additional fix (Jess, completed May 24): batch export code path now routes to the read replica endpoint. Tested with a simulated NovaTech-sized export — completed in 3 minutes 47 seconds with peak connection usage of 12.

PREVENTION:
- Added Datadog alert for connection pool utilization >60%
- Added per-account query concurrency limits
- Batch export queries now run against the read replica, not the primary
- Added Datadog dashboard panel showing per-account connection counts (so we can spot monopolization before it hits the alert threshold)
- Jess added a circuit breaker to the export partitioner: if estimated partition count exceeds 25, the export is automatically queued for off-peak execution (runs between 02:00-06:00 ET)

CUSTOMER IMPACT:
- NovaTech Labs notified, they understood. Their batch export was re-submitted after the fix and completed in 4 minutes.
- No other customers filed support tickets about the degradation.
- No SLA violations (we don't have formal SLAs yet, but if we did, 22 minutes at degraded performance would be within typical SLA bounds).
- Amir sent a brief summary to Nadia in case any customers mention it during sales conversations. Nadia confirmed none of the affected accounts are in active renewal or expansion discussions.

SECTION 3: BUG REPORTS — DETAIL

BUG-001: CSV export truncates column names longer than 64 characters
- Ticket: TK-2025-0173
- Reported by: Verdant Agriculture (their Snowflake schema uses descriptive column names like "average_soil_moisture_percentage_by_county_rolling_30d")
- Severity: Low
- Affected: 1 user
- Status: Fixed (May 24)
- Root cause: CSV writer was using a fixed-width buffer for headers. The buffer was allocated as a char[64] array in our CSV serialization module (a legacy pattern from when the module was ported from a C library). When a column name exceeded 64 characters, it was silently truncated without any error or warning.
- Fix: Replaced fixed-width buffer with dynamic string allocation. Also added a unit test to verify column names up to 512 characters. Priya ran the regression suite — all 247 tests pass. Shipped in release 2.1.3.

BUG-002: Streaming subscription drops after exactly 24 hours
- Ticket: TK-2025-0178, TK-2025-0182, TK-2025-0190
- Reported by: 3 users (Clearpoint Financial, NovaTech Labs, Meridian Data Services)
- Severity: Medium
- Affected: 3 users
- Status: Fixed (May 25)
- Root cause: WebSocket connection had a hardcoded 24-hour TTL from the load balancer configuration. Specifically, our AWS ALB had idle_timeout.timeout_seconds set to 86400 (exactly 24 hours). When the timeout was reached, the ALB terminated the WebSocket connection. Our client-side SDK did not have automatic reconnection logic — when the connection dropped, it raised a ConnectionClosedError and stopped.
- Fix details: Changed the ALB timeout to 604800 (7 days). Added automatic reconnect logic to the SDK: when a WebSocket connection drops, the SDK waits 1 second, then attempts to reconnect with exponential backoff (1s, 2s, 4s, max 30s). On successful reconnect, the SDK re-subscribes to all previously active subscriptions and replays any events that occurred during the disconnection window (using the last_event_id cursor). The reconnect logic also fires a callback so users can handle the reconnection event in their application code.
- SDK changes shipped in v0.9.2 (same release as BUG-003 fix). Server-side ALB change deployed May 25 at 05:30 ET during maintenance window.
- Priya's testing notes: Verified reconnection by simulating ALB termination in staging. SDK reconnected within 2 seconds and resumed event delivery with no data loss. Tested with 50 concurrent WebSocket connections — all reconnected successfully.

BUG-003: Python SDK throws uncaught exception on empty query result
- Ticket: TK-2025-0181, TK-2025-0188
- Reported by: 2 users (one via Intercom, one via Slack community)
- Severity: Medium
- Affected: 2 users
- Status: Fixed, SDK v0.9.2 published to PyPI
- Root cause: SDK expected at least one row in response, didn't handle empty result set. Specifically, the QueryResult class constructor called self.rows[0] to extract column metadata, which raised an IndexError when the result set was empty. This is a common pattern in data queries — users frequently run queries with filters that match no records, especially during initial setup when their data pipelines haven't fully synced yet.
- Fix: Added an explicit check for empty result sets. QueryResult now has an is_empty property and the column metadata is extracted from the response headers (which are always present) rather than from the first row. Also added a rows property that returns an empty list instead of None for empty results, since several users' code was doing `if result.rows:` which would have thrown a different error.
- Priya wrote 6 additional test cases covering empty results, single-row results, and results with null values in various positions. All tests pass.

BUG-004: Rate limit counter doesn't reset at midnight UTC as documented
- Ticket: TK-2025-0184
- Reported by: UrbanGrid Analytics (developer integrating via API noticed they were being rate-limited at 19:00 UTC despite the docs saying the counter resets at 00:00 UTC)
- Severity: Low
- Affected: 1 user (hitting rate limits at unexpected times)
- Status: Fixed (May 24)
- Root cause: Timer was using server local time (ET) not UTC. The rate limiter was implemented using a Redis EXPIREAT command with a timestamp calculated from Python's datetime.now() instead of datetime.utcnow(). Since our servers are in us-east-1 and configured for Eastern Time, the reset was happening at 00:00 ET (04:00 UTC in winter, 05:00 UTC in summer) instead of 00:00 UTC.
- Fix: Changed to datetime.utcnow() and added a comment explaining the UTC requirement. Also added the X-RateLimit-Reset response header (which we documented but weren't actually sending) so API consumers can programmatically determine when their quota resets.
- Corrected the developer docs to clarify that rate limits reset at 00:00 UTC and added the X-RateLimit-Reset header to the API reference.

BUG-005: HubSpot connector misses custom properties with spaces in names
- Ticket: TK-2025-0198
- Reported by: Cascade Ventures (they have HubSpot custom properties like "Lead Source Detail" and "Annual Contract Value")
- Severity: Low
- Affected: 1 user
- Status: In progress — fix expected May 27
- Root cause: URL encoding issue in the HubSpot API client. When querying the HubSpot CRM API for custom properties, we build the URL with property names as query parameters. Property names containing spaces were not being URL-encoded, resulting in the API returning a 400 error for those specific properties. The connector silently skipped properties that returned errors (this is intentional for handling deprecated properties) but the side effect was that valid properties with spaces were also being skipped.
- Proposed fix (Jess): Apply urllib.parse.quote() to all property names in the query parameter builder. Also add a warning log when a property fetch returns a non-200 status so that silent skipping is at least visible in the connector logs.
- Additionally, Amir flagged that the "silent skip" behavior should be reconsidered. Silently skipping errors is convenient for deprecated properties but can mask real issues like this one. Proposed compromise: skip with a warning log for 404 (property not found / deprecated), but raise for other error codes. Jess agreed, implementing both changes together.

SECTION 6: WEBHOOK RELIABILITY ANALYSIS

Current architecture: direct HTTP POST on event trigger. If the POST fails, we retry 3 times with exponential backoff (1s, 5s, 30s). If all retries fail, the event is dropped.

Problem: Under high load, the synchronous POST blocks the event processing pipeline. At 10K events/minute, the blocking causes a cascade that drops ~2% of events.

DETAILED ANALYSIS:

The event processing pipeline currently handles webhook delivery inline. When an event occurs (e.g., a connector sync completes, a data quality rule triggers), the event handler constructs the webhook payload and immediately attempts an HTTP POST to the customer's configured endpoint. This POST is synchronous — the event processing thread blocks until the POST completes or times out (30 second timeout).

Under normal conditions (< 1K events/minute), this works fine. The average webhook delivery time is 180ms, and the pipeline has 10 worker threads, giving us a theoretical throughput of approximately 3,300 deliveries per minute.

Under stress conditions (10K events/minute), the pipeline saturates. If even a small percentage of customer webhook endpoints are slow (>1 second response time) or temporarily unavailable, the worker threads get backed up. When all 10 threads are blocked on slow deliveries, new events queue in memory. The in-memory queue has a 10,000 event cap. When the cap is reached, new events are dropped. In our stress test, approximately 200 of 10,000 events per minute were dropped (the 2% figure).

The 2% drop rate is unacceptable for customers building automated workflows on webhook events. Meridian Data Services specifically mentioned in TK-2025-0180 that they want to use webhooks to trigger their data pipeline — dropped events would mean missed pipeline runs.

Proposed fix: Move to asynchronous queue-based delivery.
1. Events are written to an SQS queue immediately (non-blocking). The event handler writes the payload to SQS and returns immediately, freeing the worker thread. SQS write latency is typically <10ms, so this adds negligible overhead to the event processing pipeline.
2. A separate worker pool reads from the queue and delivers webhooks. This pool can be scaled independently of the event processing pipeline. Initial plan: 20 workers, which gives us a theoretical throughput of approximately 6,600 deliveries per minute assuming 180ms average delivery time. If we need more, we can scale to 50 workers without additional infrastructure.
3. Failed deliveries are retried with exponential backoff up to 24 hours. Retry schedule: 1 minute, 5 minutes, 30 minutes, 2 hours, 8 hours, 24 hours. Each retry is a separate SQS message with the retry count in the message attributes.
4. Dead letter queue for events that fail all retries. Events in the DLQ are retained for 14 days. We'll add a webhook delivery log in the UI so customers can see failed deliveries and manually retry them.

Additional consideration (from Marcus): we should also add webhook signature verification so customers can validate that webhooks are genuinely from Verdana. Proposed approach: HMAC-SHA256 signature in the X-Verdana-Signature header, using a per-account webhook secret that customers can rotate via the API. Amir will include this in the implementation.

This gives us at-least-once delivery guarantees. Estimated implementation time: 3 days.

Implementation breakdown:
- Day 1: SQS queue setup, event handler modification to write to queue, worker pool scaffold
- Day 2: Retry logic, dead letter queue, webhook signature generation, delivery logging
- Day 3: Testing (load test at 10K events/min, failure injection, signature verification), deployment

Marcus approved the approach. Starting May 26.
