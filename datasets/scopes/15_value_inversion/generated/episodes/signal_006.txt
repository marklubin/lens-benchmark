PRIORITY ITEMS:

1. Dashboard Sprint Kickoff — Jess and Lina to begin Q2 dashboard work. Target: have the data binding layer designed and prototyped by end of sprint. Ship target remains July 15.

   DETAILS:
   - Jess to architect the data binding layer — how dashboard widgets will subscribe to query results

   - Jess noted that the data binding layer is the hardest part — if we get that right, the UI layer is relatively straightforward. She estimated 3-4 days for the binding layer prototype and 2 days for Lina to build a basic proof-of-concept UI on top of it.

2. Webhook Reliability — Amir to fix the 2% drop rate under load. Target: <0.1% drop rate.

   DETAILS:
   - Current webhook delivery pipeline: API receives event → pushes to SQS queue → Lambda consumer → HTTP POST to customer endpoint
   - The 2% drop rate is occurring between SQS and Lambda — messages are being consumed but the Lambda is timing out on some deliveries
   - Amir's proposed fix: increase Lambda timeout from 10s to 30s, add a dead-letter queue for failed deliveries, implement automatic retry with exponential backoff (3 attempts, 1s/5s/30s delays)
   - Secondary issue: some customer webhook endpoints are slow (>5s response time), which compounds the timeout problem
   - Amir estimated 2-3 days for the fix plus testing

3. Onboarding Improvement — Lina to redesign the first-run experience. Add a guided setup wizard that helps non-technical users configure their first connector and run their first query without reading API docs.

   DETAILS:
   - Current first-run experience: user signs up → lands on an empty dashboard with a "Get Started" link to API docs
   - Proposed wizard flow: (a) select your first data source from a visual grid, (b) OAuth or credential entry with inline validation, (c) run a sample query with pre-built templates, (d) see results in a basic table view
   - Lina estimated 4-5 days for the wizard implementation
   - Tomoko to provide design mockups by Monday May 26
   - Derek asked whether the wizard should also cover non-OAuth connectors (PostgreSQL, MySQL, Snowflake). Lina said yes — those are the ones that need the most hand-holding. The wizard will include inline help text and validation for connection strings, SSL settings, and IP whitelisting instructions.

4. Documentation Gaps — Amir to add 5 missing API examples (batch export with filters, streaming subscription management, multi-connector queries, webhook configuration, SDK quickstart for JavaScript).

   DETAILS:
   - These were the top 5 documentation requests from support emails and in-app feedback this week
   - Amir to write the examples and submit them as a PR to the docs repo
   - Each example should include: curl command, Python SDK equivalent, JavaScript SDK equivalent, expected response, and common error scenarios
   - Estimated effort: 1.5 days total
   - Nadia asked Amir to also update the Snowflake connector docs with clearer instructions on warehouse/role/schema configuration, given the 71% first-attempt success rate. Amir agreed — will add a troubleshooting section.

5. Connector Requests — Evaluate Jira and Asana connectors based on user requests. Scoping only this sprint.

   DETAILS:
   - 6 users have requested Jira integration, 3 have requested Asana
   - This sprint is scoping only — Amir to evaluate the Jira and Asana APIs, estimate build effort, and write a one-page proposal
   - No build work will happen this sprint
   - Derek noted that Jira's API has rate limiting that could be tricky for real-time sync. Amir said he'd investigate whether we'd need to use Jira webhooks (push) vs. polling (pull).
