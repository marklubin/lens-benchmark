SECTION 3: SPRINT PLANNING — DASHBOARD SPRINT 1 (MAY 28 - JUNE 11)

GOAL: Complete the data binding layer prototype. This is the core technical challenge — connecting dashboard widgets to live Verdana API queries with automatic refresh.

OWNER: Jess Thornton (engineering), Lina Choi (design)

CONTEXT (where we left off):
- Grid system: DONE (from pre-launch work)
- Chart rendering: DONE (bar, line, area, pie, number, table widgets)
- Widget library: DONE (Lina's designs)
- Data binding layer: NOT STARTED
- Dashboard management UI (create, edit, share, permissions): NOT STARTED

Technical notes from Jess on the data binding layer:
- The grid system uses a 12-column CSS grid with snap-to-grid drag-and-drop. Each widget occupies a rectangular region defined by (col_start, row_start, col_span, row_span). This is already implemented and working.
- Chart rendering uses Recharts (React charting library) with a custom theme layer that matches our design system. Bar, line, area, pie, number (single KPI), and table widgets are all implemented with sample data. They just need to be connected to live data.
- The data binding layer needs to: (1) accept a Verdana API query definition (endpoint, parameters, filters, aggregations), (2) execute the query against the API, (3) transform the response into the shape expected by each widget type, (4) handle refresh on a configurable interval, (5) handle loading states and errors gracefully.
- Jess estimates 4-5 days for the core data binding layer, 3-4 days for the query builder UI. Total: 8-9 working days, which fits in the 2-week sprint with a day or two of buffer.

THIS SPRINT:
1. [JESS] Build the data binding layer — a widget connects to a Verdana API query and auto-refreshes on a configurable interval (1s, 5s, 30s, 1m, 5m)
   - Sub-tasks:
     a. Define the QueryBinding schema (API endpoint, params, refresh interval, transform function)
     b. Implement the QueryExecutor service (handles auth, rate limiting, error retry)
     c. Build the DataTransformer layer (maps API responses to widget-specific data shapes)
     d. Wire up refresh loop with configurable intervals and manual refresh button
     e. Implement loading skeleton states and error display for each widget type

2. [JESS] Implement query builder UI — visual interface for constructing API queries without writing JSON
   - Sub-tasks:
     a. Connector selector (which data source)
     b. Metric selector (which fields to display)
     c. Filter builder (date range, dimension filters)
     d. Aggregation selector (sum, avg, count, min, max, by time period)
     e. Preview panel (shows live results as you build the query)

3. [LINA] Finalize the dashboard creation flow — create dashboard → add widgets → configure data source → arrange layout → save → share
   - Sub-tasks:
     a. Empty state design (first dashboard creation experience)
     b. Widget picker modal (browse and select widget types)
     c. Data source configuration panel (integrates with Jess's query builder)
     d. Dashboard toolbar (save, share, settings, fullscreen)
     e. Responsive breakpoint designs (desktop, tablet — mobile deferred to Sprint 3)

TARGET FOR END OF SPRINT:
