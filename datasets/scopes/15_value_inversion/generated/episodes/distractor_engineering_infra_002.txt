SECTION 2: CI/CD IMPROVEMENTS — IN PROGRESS

I've identified four improvements to the pipeline, ranging from quick wins to longer-term investments. Prioritized by impact and effort.

IMPROVEMENT 1: PARALLEL TEST EXECUTION

Current: All 142 tests run sequentially in a single process. This is the single biggest contributor to CI time.

Proposed: Split tests into 4 parallel groups using pytest-xdist. The split is based on test file — xdist distributes test files across workers rather than individual tests, which avoids issues with test-level fixtures.

Technical details:
- Adding `pytest-xdist` to dev dependencies
- CI step changes from `uv run pytest tests/` to `uv run pytest tests/ -n 4`
- GitHub Actions runner has 4 vCPUs so 4 workers is the natural fit
- Need to verify no test interdependencies (shared DB state, file locks, etc.)
- Integration tests use separate testcontainer instances per worker — verified this works locally

Expected impact: Test step drops from 3m 22s to ~1m 10s.
Status: PR #87 open, testing on a feature branch. Amir found one test (`test_connector_sync_ordering`) that relied on test execution order — fixed by adding explicit setup in the test. Running the full suite with `-n 4` locally 10 times, all green.

IMPROVEMENT 2: DOCKER LAYER CACHING

Current: Docker builds use GitHub Actions cache, but the cache hit rate is only 60% because our requirements.txt changes frequently. Every time someone adds or bumps a dependency, the entire `pip install` layer is invalidated.

Proposed: Use a two-stage Dockerfile — base image with dependencies (cached for weeks) and app image with source code (rebuilt each time). The base image is tagged with a hash of `uv.lock` so it only rebuilds when dependencies actually change.

```
# Stage 1: Base (cached unless uv.lock changes)
FROM python:3.12-slim AS base
COPY uv.lock pyproject.toml ./
RUN uv sync --no-dev

# Stage 2: App (rebuilt each time)
FROM base AS app
COPY src/ ./src/
COPY alembic/ ./alembic/
```

The CI workflow computes `sha256(uv.lock)` and checks ECR for a base image with that tag. If it exists, Docker pulls it and only builds stage 2. If not, it builds both stages and pushes the base image.

Expected impact: Docker build drops from 1m 12s to ~25s for most builds. Full rebuilds (when dependencies change) will be ~1m 30s due to the extra push step, but these happen maybe once or twice a week.
Status: Implemented, merged in PR #85. Cache hit rate since merge: 92% (up from 60%).

IMPROVEMENT 3: MYPY INCREMENTAL MODE

Current: mypy runs a full check every CI run. As our codebase grows, this will get slower.

Proposed: Enable mypy incremental mode with persistent cache. The cache is stored as a GitHub Actions cache artifact keyed on the mypy version + Python version + commit SHA of the base branch. On PR runs, we restore the cache from the base branch and run mypy incrementally — it only rechecks files that changed.

Technical details:
- Cache key: `mypy-{mypy_version}-{python_version}-{base_branch_sha}`
- Fallback key: `mypy-{mypy_version}-{python_version}-` (latest cache from any commit)
- mypy cache directory: `.mypy_cache/` — stored and restored between runs
- Incremental mode is mypy's default, but without a persistent cache it's useless in CI (each run starts fresh)
- Risk: cache corruption could cause false negatives (mypy misses a real error). Mitigation: weekly scheduled CI run with `--no-incremental` to catch any drift.

Expected impact: Type check drops from 1m 8s to ~20s for most runs. Full check (~1m 8s) on cache miss.
Status: Testing — need to validate cache correctness. PR #89. I've been running both modes side-by-side on feature branches for the past 3 days. Incremental mode has reported the same results as full mode in all 14 runs so far.

Priya raised a concern about `--follow-imports=skip` interacting badly with incremental mode. I tested this — we don't use `--follow-imports=skip` anywhere in our mypy config, so it's not an issue. But good catch.

IMPROVEMENT 4: DEPLOY PREVIEW ENVIRONMENTS

Current: Developers test locally or on staging. No per-PR preview environments. The staging environment is shared, which means only one PR can be on staging at a time. We've had a few scheduling conflicts where Amir needed staging for a demo while Priya was testing a migration.

Proposed: Spin up an ephemeral ECS service for each PR with a unique URL (e.g., `pr-87.preview.verdana-internal.dev`). Automatically teardown when PR is closed or after 48 hours of inactivity. Each preview gets its own ECS task and connects to a shared preview RDS instance (separate from staging and production).

Architecture:
- GitHub Actions workflow triggered on PR open/sync
- Terraform module creates: ECS service (1 task), ALB target group, Route53 record
- Preview database: shared RDS instance (`preview-db`) with per-PR schema (schema name = `pr_{number}`)
- Kafka: shared preview cluster with per-PR topic prefix
- Redis: shared preview instance with per-PR key prefix
- Teardown: GitHub Actions workflow triggered on PR close, or scheduled cleanup Lambda runs every 6 hours and removes previews with no traffic for 48 hours

Expected impact: Better testing, fewer staging conflicts, ability for product/design to review before merge. Hannah (product) has been asking for this — she wants to review UI changes without cloning the repo and running locally.
Estimated cost: ~$15/PR/day (ECS + RDS connection). At our current volume (~3 open PRs at any time), that's ~$45/day or ~$1,350/month. This is significant relative to our total infra cost.
Status: Scoping phase — estimated 2 weeks of Kevin's time. Deprioritized in favor of more urgent work (DR failover testing, parallel CI). Targeting Q3 if the team decides the cost is justified.

Marcus's note: "I like the idea but want to see us hit 200 accounts before we invest in preview environments. At our current pace the staging conflicts are annoying but manageable. Revisit in July."

FINDING 1: REDIS CLUSTER IS OVER-PROVISIONED

Current: 3-shard Redis Cluster with 1 replica per shard = 6 nodes × cache.r6g.large (13.07 GB RAM, 2 vCPU each)
Current cost: $1,104/month
Current utilization:
- Memory: 3.2% (2.5 GB used out of 78.4 GB total across cluster)
- CPU: 0.8% average, 2.1% peak
- Network: 12 Mbps average, 45 Mbps peak
- Connections: 24 average, 68 peak
- Operations: 1,200 ops/sec average, 3,800 ops/sec peak
- Cache hit rate: 94.6%

This is dramatically over-provisioned. At our current scale (112 accounts), a single cache.r6g.medium node (6.38 GB RAM) would handle our workload with room to grow to 500+ accounts. The 3-shard cluster was provisioned for a projected 10,000-account workload that we're nowhere near.

However: Marcus and I agreed pre-launch to keep the cluster to avoid a migration later. I still think that was the right call, but I want to flag the cost. Redis cluster migrations involve downtime and application changes (different connection handling for cluster vs. standalone mode).

Detailed cost comparison:

| Configuration | Nodes | Monthly Cost | Capacity | When to Use |
|:---|:---|:---|:---|:---|
| Current (3-shard cluster) | 6 × r6g.large | $1,104 | ~10,000 accounts | If we grow fast |
| Downsized cluster | 3 × r6g.medium | $552 | ~3,000 accounts | 200-3,000 accounts |
| Standalone + replica | 2 × r6g.medium | $368 | ~1,000 accounts | 200-1,000 accounts |
| Standalone (no replica) | 1 × r6g.medium | $184 | ~500 accounts | If we accept HA risk |

Alternative: Downgrade to a 3-node cluster with cache.r6g.medium nodes
Savings: $1,104 → $552/month = $552/month saved
Risk: If we grow faster than expected, we'd need to scale the cluster, which requires downtime. Scaling from 3 medium nodes to 6 large nodes requires a blue-green migration (~2 hours of engineering time + 15 minutes of downtime).

RECOMMENDATION: Keep current setup until we reach 200 accounts, then re-evaluate. If growth is slower than projected, downgrade in Q3. The $552/month savings is real but not urgent — it's about 25% of total infra cost.

Marcus's note: "Agreed. Flag for Q3 review. I'd rather over-pay for headroom than deal with a Redis migration while we're trying to close deals."

FINDING 2: ECS FARGATE MINIMUM TASKS

Current: Minimum 2 tasks, scaling to 16. Each task: 1 vCPU, 2 GB RAM.
Current cost: ~$198/month (based on actual usage with autoscaling)

Observation: Between midnight and 6 AM ET, traffic drops to near zero. We're paying for 2 idle tasks during those 6 hours.

Traffic pattern (requests/minute):
- 12 AM - 6 AM ET: 0-2 req/min
- 6 AM - 9 AM ET: 15-40 req/min (ramp up)
- 9 AM - 6 PM ET: 50-120 req/min (business hours)
- 6 PM - 12 AM ET: 10-30 req/min (wind down)

Alternative: Scale to 0 tasks overnight with a cold-start Lambda that wakes the service on first request. The Lambda would sit behind the ALB and, on receiving a request, trigger ECS scale-up and return a "warming up" response. Subsequent requests hit the ALB health check loop until the task is healthy.

Savings: ~$35/month (2 tasks × 6 hours/day × 30 days × Fargate pricing)
Complexity: New Lambda function, ALB listener rule changes, cold-start handling in client SDKs
Risk: Cold start latency (14 seconds) for first request after overnight. This includes ECS task provisioning (8s), container startup (4s), and application startup (2s).

RECOMMENDATION: Not worth the complexity. $35/month savings doesn't justify cold start UX impact or the engineering effort to build and maintain the scale-to-zero mechanism. If any customer's automated systems make early-morning API calls, they'd hit a 14-second delay that could cause timeouts.

Amir asked about this in standup — he agreed it's not worth it. "If even one customer has a cron job at 5 AM that times out because we wanted to save $35, that's a bad trade."

| 9 | Event insert (batch 1000) | 35ms | 75ms | 110ms | 300/day | Kafka consumer writes |
| 10 | Simple SELECT by ID | 1ms | 3ms | 8ms | 5,000/day | API key validation, etc. |

ANALYSIS:
- None of these are concerning at current scale. The slowest query (340ms p95) is well within acceptable limits. Our API SLA target is 500ms p95, and even the heaviest aggregation query is under that.
- The connector sync status check (#8) is called frequently and could benefit from Redis caching. Filed as a ticket (JIRA-247). The sync status only changes when a sync completes (every 15 minutes), so caching with a 60-second TTL would eliminate ~95% of these queries. Amir is picking this up for his June sprint.
- Event batch inserts (#9) are efficiently using COPY command. No optimization needed. We tested INSERT vs COPY during pre-launch and COPY was 4x faster for batch sizes >500.
- Full-text search (#2) will become a problem at scale. The current implementation uses PostgreSQL's built-in `tsvector` with a GIN index, which works fine for 2.4M rows but will degrade at 50M+. We should evaluate pgvector or Elasticsearch for search workloads in Q3. I've created a research ticket (JIRA-248) for Priya to investigate options.
- The cross-connector JOIN (#3) is the most architecturally interesting query. It joins the `events` table with itself across different `connector_id` values within the same account. At scale, this will need a materialized view or pre-computed correlation table. Not urgent at 8 queries/day.
