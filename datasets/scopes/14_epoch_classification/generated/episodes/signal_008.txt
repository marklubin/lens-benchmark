[2025-04-19 14:30:22] priya.kaur
i want to document our new deployment process before we start using staging on monday. writing it up now. here's the draft:

---
**Foldly Deployment Process v1.0**

**Effective: 2025-04-21**

1. Create a feature branch from `main`
2. Make changes, commit to branch
3. Open a Pull Request against `main`
4. Automated tests run on PR (CI pipeline)
   - All tests must pass before merge is allowed
5. At least 1 teammate must review and approve the PR
   - For billing/auth/data changes: 2 approvals required
6. Merge PR to `main`
7. Merge triggers automatic deployment to staging
8. Engineer verifies the change on staging
   - For billing changes: test with Stripe test mode
   - For auth changes: test SSO flow
   - For document changes: test upload/download/search
9. Engineer promotes from staging to production
   - Promotion requires passing staging verification
10. Monitor production for 30 minutes post-deploy
    - Check Sentry for new errors
    - Check Grafana for anomalies (when available)

**Emergency Hotfix Process**:
- Same flow, but reviewer has 15-minute SLA
- Staging verification can be abbreviated
- Post-deploy monitoring extended to 1 hour
- Incident channel stays open until monitoring period ends

**Never**:
- Deploy from a local machine directly to production
- Merge without tests passing
- Deploy without at least 1 code review
---
