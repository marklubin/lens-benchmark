## Timeline

**06:12 UTC** — Nara Logistics support contact emails support@foldly.io: "We are seeing duplicate charges on our company credit card. Multiple invoices for the same period. Please investigate urgently." Email forwarded from the Nara Logistics accounts payable team, referencing Stripe receipt IDs for the duplicate transactions. The email includes a screenshot of their credit card statement showing eight line items from Foldly with identical descriptions but different transaction timestamps.

**06:42 UTC** — Ravi Mehta sees the email, escalates to #engineering. Creates #incident-20250413. Tags the incident as SEV-1 per the internal severity rubric: "Any incident that results in incorrect financial transactions affecting customer accounts." Ravi pulls up the Nara Logistics account in the admin panel to verify their current subscription status and most recent invoicing records.

**06:45 UTC** — Marcus Webb begins investigating. Checks Stripe dashboard. Navigates to the Events tab and filters by `invoice.paid` events for the past 24 hours. Notices an unusual clustering of events between 23:40 and 00:15 UTC the previous night.

**06:58 UTC** — Marcus confirms: 14 duplicate invoices detected across 3 enterprise customers. Nara Logistics: 8 duplicate charges totaling $8,247. Belmont Partners: 4 duplicate charges totaling $3,118. Crestview Capital (trial account): 2 duplicate charges totaling $892. **Total duplicate charges: $12,257.** Marcus documents each duplicate by Stripe invoice ID and charge ID in a spreadsheet that he drops into the incident channel.

**07:15 UTC** — Root cause identified by Marcus: The webhook handler in billing.py processes `invoice.paid` events and updates subscription status. A Stripe API timeout on April 12 at ~23:40 UTC caused the webhook delivery to retry. The idempotency check was supposed to prevent reprocessing, but the idempotency key was based on the Stripe event ID — and Stripe regenerated the event ID for each retry attempt, bypassing the dedup logic. Marcus traces the code path: the `handle_invoice_paid()` function in billing.py receives the webhook payload, extracts `event["id"]`, checks it against the `processed_webhook_events` set in the database, and if not found, proceeds to create a new invoice record and initiate a Stripe charge. The flaw is that `event["id"]` is unique per delivery attempt, not per business event.

**07:22 UTC** — Priya Kaur joins the incident. Asks Marcus to stop the bleeding immediately. Priya reviews the Stripe webhook delivery logs in the Stripe dashboard to confirm that the retry behavior is consistent with what Marcus has described. She notes that Stripe's retry schedule typically follows an exponential backoff pattern: immediate, ~30s, ~5m, ~30m, etc.

**07:30 UTC** — Marcus deploys hotfix: webhook handler now checks the `invoice.id` field (which is stable across retries) instead of the event ID. Additionally adds a check against the `invoices` table to skip processing if an invoice with the same Stripe invoice ID already exists. The diff is approximately 15 lines of code: replacing the `event_id` variable reference on line 412 with `event["data"]["object"]["id"]`, and adding a secondary check that queries the invoices table (`SELECT 1 FROM invoices WHERE stripe_invoice_id = ?`). Marcus runs the change through a quick local test with a mocked webhook payload before deploying.

**07:45 UTC** — Hotfix verified in production. No further duplicate processing observed. Marcus watches the webhook logs for 15 minutes to confirm that incoming retries are correctly being deduplicated by the new invoice ID check. Three retries come in during that window and all are properly rejected with a 200 OK response (to tell Stripe to stop retrying).

**08:10 UTC** — Ravi Mehta begins customer communication. Calls Nara Logistics CFO directly. Explains the situation, confirms refunds will be processed within 24 hours. Ravi prepares a templated email for the other two affected customers with the same basic information: a brief description of the bug, confirmation that refunds have been initiated, and a timeline for when funds will appear back in their accounts. He customizes each email with the specific dollar amount and number of duplicate charges for that customer.

**09:30 UTC** — All refunds initiated via Stripe dashboard. Estimated 3-5 business days for funds to return to customer accounts. Marcus initiates each refund individually through the Stripe dashboard, selecting the original charge and issuing a full refund. Stripe's refund confirmation emails are sent automatically to each customer's billing contact. Ravi follows up with a personal email to each customer confirming the refund initiation.

## Root Cause

1. **Idempotency bug**: Webhook handler used Stripe event ID as the deduplication key. Stripe regenerates event IDs on retry, so retried webhooks bypassed the dedup check. The code on line 412 of billing.py checked `if event_id in processed_events` where `event_id` was sourced from `event["id"]` (the delivery-specific event identifier) rather than `event["data"]["object"]["id"]` (the stable invoice identifier). Stripe's webhook best practices documentation explicitly warns against using event IDs for idempotency.

2. **DB contention**: Scheduled database backup overlapped with the usage reporting cron job, causing write lock contention. An incoming webhook timed out waiting for the lock, triggering Stripe's retry mechanism. The backup used `VACUUM INTO` which acquires an exclusive lock on the SQLite database file. The backup duration had grown from ~5 minutes (when billing launched with 15MB of data) to 41 minutes (with 340MB of data) due to accumulated webhook event log payloads. The 30-second webhook handler timeout was exceeded while waiting for the lock to release.

3. **No test coverage**: The idempotency logic was never tested. A basic unit test with simulated retries would have caught this bug. The billing module was merged (PR #247) with zero test files in the diff.

4. **No code review**: The billing module was written and deployed by a single developer with no peer review. The initial review was acknowledged as a rubber stamp due to time pressure to begin generating revenue.

5. **No staging environment**: There was no way to test webhook retry behavior in a non-production setting. All webhook integration testing was done directly in production.

6. **No proactive monitoring**: No alerting was configured to detect duplicate invoice creation. The incident was discovered 7 hours after it occurred, only because a customer noticed the duplicate charges on their credit card statement and emailed support.
