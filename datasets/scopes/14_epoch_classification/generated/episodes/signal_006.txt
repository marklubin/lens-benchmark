## 1. Timeline Review

Priya shared her screen and walked through the incident timeline from the report,
starting from the first customer notification through resolution. She went through
each timestamp methodically, pausing for discussion at key decision points.

**Priya**: "Let's start with the timeline. I'm pulling up the incident report now.
Everyone should have this in the #incident-20250413 channel from last night. Marcus,
you wrote most of this — want to walk us through the technical sequence?"

**Marcus**: "Sure. So the short version is that DB contention from the analytics
cron job caused webhook processing to slow down. Stripe's webhook delivery has a
retry mechanism — if we don't respond within their timeout window, they resend the
event. When our handler processed the retried events, it treated them as new events
because of the idempotency bug. The idempotency bug was straightforward — I used the
event ID instead of the invoice ID. Stripe's docs actually warn about this but I
missed it. I was moving fast and didn't read the retry behavior section carefully."

**Danny**: "Can you clarify the event ID vs. invoice ID thing? I want to make sure I
understand the mechanics."

**Marcus**: "Yeah. So when Stripe sends a webhook, each delivery attempt has a unique
event ID — evt_xxxxx. But if the same event gets retried, Stripe generates a new
event ID for the retry. So if you're using the event ID as your idempotency key,
the retry looks like a completely new event. What you're supposed to use is the
invoice ID — inv_xxxxx — which stays the same across retries. I used evt_ instead
of inv_, so every retry got treated as a fresh charge."

**Lina**: "And for context, the DB contention is worth understanding but it's not
the root cause. The analytics cron job was running its nightly aggregation query,
which takes a full table lock on the transactions table. That caused the webhook
handler's DB writes to block, which meant we were slow to respond to Stripe, which
triggered the retries. But — and this is important — the DB contention was a
contributing factor but not the root cause. Even without the contention, any network
hiccup or server restart would have triggered retries and hit the same bug."

**Priya**: "So the contention was the trigger, but the idempotency bug was the
vulnerability."

**Lina**: "Exactly. The contention exposed the bug, but the bug would have been
triggered eventually by anything that caused a retry."

**Marcus**: "Right. And I should say — the cron job contention is its own issue.
That analytics query shouldn't be taking a full table lock. But Lina's right,
even if we fix the contention, the idempotency bug would still be sitting there
waiting for the next trigger."

**Jess**: "The real question isn't why the bug existed — bugs happen. The question
is why nothing caught it before it hit production and affected customers."

**Danny**: "That's what I keep coming back to. This is a pretty standard webhook
integration pattern. Any test that simulated a retry would have caught it immediately."

**Priya**: "Agreed. That's what we need to dig into."

**Lina**: "Before we move on from the timeline — I want to note that the detection
was also slow. The first customer email came in at 22:47 UTC on the 13th, but we
didn't start investigating until Marcus saw it the next morning. There's no alerting,
no anomaly detection on billing events, nothing."

**Priya**: "Good point. Let's capture that as a systemic issue. Ravi, do you have
the customer impact numbers finalized?"

**Ravi**: "Yes. Three accounts were affected. Nara Design Studio was double-charged
$4,200. Belmont Partners was double-charged $5,800. And Crestview Analytics was
double-charged $2,000. Total duplicate charges: $12,000. All refunds have been
processed as of this morning."

**Priya**: "OK. Let's move to the 5 Whys."

## 2. Root Cause Analysis — 5 Whys

Priya facilitated the 5 Whys exercise on a shared whiteboard. Each "why" was
discussed and agreed upon by the group before moving to the next level.

**Why did customers get double-charged?**
Because the webhook handler processed duplicate events.

**Priya**: "This one is straightforward. Marcus, anything to add?"

**Marcus**: "No. The handler saw what it thought were two separate events and
processed both. Two charges per invoice."

---

**Why did the handler process duplicate events?**
Because the idempotency check used the event ID, which Stripe regenerates on retries.

**Danny**: "Is there any logging that shows the duplicate processing? Like, can we
see the two event IDs that mapped to the same invoice?"

**Marcus**: "Yeah, I pulled the logs last night. For the Nara charge, we processed
evt_3kN9xR2eZvKY at 22:31 and evt_7mP4hT8nWqBX at 22:34. Both mapped to
inv_Q4n8vK2mXp. The second event was Stripe's retry after we timed out on the first."

**Lina**: "Three-minute gap between events. That lines up with the DB contention
window from the cron job."

---

**Why wasn't this caught during development?**
Because there are no automated tests for the billing module. Manual testing only
covered the happy path — a single webhook delivery without retries.

**Jess**: "I want to dig into this one. Marcus, when you were building the billing
integration, did you test webhook retries at all?"

**Marcus**: "Honestly, no. I tested a single webhook delivery end-to-end — Stripe
sends event, we process it, charge goes through. That worked. I didn't test the
retry scenario because I didn't think about it. And there was no test suite that
would have forced me to think about it."

**Priya**: "Was there any documentation or checklist for what to test when building
a billing integration?"

**Marcus**: "No. There's no documentation at all, really. I was working from Stripe's
docs and my own judgment."

**Danny**: "Which is fine for a prototype, but this went straight to production
handling real money."

---

**Why are there no automated tests?**
Because the team has prioritized shipping speed over testing. Testing has been
repeatedly deferred — most recently when Danny proposed writing tests and was told
to skip them and focus on shipping.

**Priya**: "Danny, can you walk us through what happened with the testing proposal?"

**Danny**: "Sure. About two weeks ago — I think it was March 31 or April 1 — I
asked in #engineering whether we should write tests for the billing code before
we onboarded enterprise customers. The response was basically 'not now, ship the
feature first, we'll add tests when we have time.' I think Marcus said something
similar. And Priya, you said to keep focused on the sprint goals."

**Priya**: "I did. I remember that conversation."

**Marcus**: "Yeah, I think I said something like 'tests would be nice but let's
get the integration done first.' Which in retrospect was exactly wrong."

**Danny**: "And that wasn't the first time. Before that, Lina raised concerns about
not having a staging environment, and the answer was the same — later, not now,
after the next milestone."

---

**Why has testing been repeatedly deferred?**
Because the team culture explicitly values speed over process. The phrase "we'll
add tests when we have time" has become a running joke, and the "testing sprint"
has been pushed to "late April" and then "mid-May" without ever materializing.

**Jess**: "I want to put a fine point on this. It's not that anyone made one bad
decision. It's that we've made the same decision — defer process, ship faster —
over and over for weeks. Each individual deferral seemed reasonable. In aggregate,
they created this situation."

**Lina**: "The 'testing sprint' thing really bothers me. It was first mentioned in
early March. Then it was 'late March.' Then 'mid-April.' Now it's 'late April.'
It keeps sliding because there's always something more urgent. But at some point,
the lack of testing becomes the urgent thing. That's where we are now."

**Priya**: "You're both right. And I want to own my role in that pattern. But first,
let's talk about the systemic issues."

## 4. Systemic Issues Identified

Priya summarized the systemic issues that emerged from the discussion. The team
reviewed and agreed on the risk levels by consensus.

| Issue | Description | Risk Level |
|-------|-------------|------------|
| No code review | All code ships without peer review | Critical |
| No automated tests | Zero test coverage across entire codebase | Critical |
| No staging environment | All deployments go directly to production | Critical |
| Monolithic files | Core logic in 2-3 massive files (routes.py: 2,247 lines, billing.py: 1,100+ lines) | High |
| No deployment process | Developers deploy from local machines at any time | High |
| No on-call rotation | No defined process for after-hours incidents | Medium |
| No documentation | Server config, cron jobs, and architecture undocumented | Medium |
| No database migration process | Schema changes applied directly to production DB | Medium |

**Lina**: "I want to add context on the monolithic files issue. routes.py is 2,247
lines in a single file. billing.py is over 1,100 lines. When everything is in one
file, there's no natural module boundary that forces you to think about interfaces
and contracts. The webhook handler, the charge creation logic, the refund logic,
the invoice generation — they're all tangled together in billing.py. That makes it
harder to test any one piece in isolation, and it makes it easier to introduce bugs
because a change to one function can have side effects on another."

**Marcus**: "Yeah, and it makes code review harder too, even once we have it. Reviewing
a diff in a 1,100-line file is way harder than reviewing changes to a focused module
with clear responsibilities."

**Jess**: "The deployment process one is also worth flagging. Right now, any of us
can deploy at any time from our local machine. There's no deploy queue, no deploy
log, no coordination. If two people deploy at the same time, we could end up in a
weird state. And there's no way to quickly roll back."

**Danny**: "We don't even have a consistent way to know what's currently running in
production. If someone asks me 'what version of billing.py is live right now,' I'd
have to SSH in and check."

**Priya**: "All noted. Let's get to the action items."

## 6. Cultural Discussion

Priya transitioned the conversation to a broader discussion about team culture and
engineering values. She explicitly asked each person to share their perspective.

**Priya**: "Before we wrap up, I want to have a broader conversation. This incident
isn't just a technical failure — it's a cultural one. I want everyone to share what

**Priya**: "I want to say something to the whole team. We built something real in
the last two weeks. 7 features, 2 enterprise customers, $60K ARR. That's incredible.
But we also shipped a bug that cost our customers $12K and damaged our reputation.

guardrails, and eventually that catches up to you."

**Priya**: "I've been thinking about this a lot since last night. I started this
company because I wanted to build something great, and I believed that speed was the
most important thing. Ship fast, iterate, figure it out. And that worked for the
first few months when it was just me and Jess and the stakes were lower. But we're
handling real money now. We have enterprise customers with legal teams. The stakes
are different, and our process needs to catch up."

**Marcus**: "I think the lesson is simpler. I wrote payments code in a single file

difference."

**Marcus**: "And I want to be accountable for that. I've been writing production code
for five years. I know better. I got caught up in the sprint energy and let my
standards slip. That's not something I want to repeat."

going to slow us down meaningfully, and it would have prevented this incident."

**Danny**: "I just want to say — I'm glad we're having this conversation. I've been
uncomfortable with the 'just ship it' thing for a while but didn't want to slow the

**Danny**: "I think part of the issue is that when you're the new person on the team

**Priya**: "Later is now. Starting today, we review code. We write tests for critical
